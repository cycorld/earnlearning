## Part 2: 구조 이해 · 권한 · 프롬프트 카탈로그 (35분)

### 3. 왜 백엔드가 필요한가? (한 번만 정리)

여러분이 지금 만든 것이 "제대로 된 백엔드 위에 올라가 있는지" 확인하는 기준입니다.

| 기능 | 의미 | 확인 방법 |
|------|------|---------|
| **저장 (Persist)** | 새로고침해도 남아 있기 | 새로고침 후 데이터가 그대로 있는가? |
| **공유 (Share)** | 다른 사람 기기에서도 보이기 | 다른 브라우저/폰에서 열었을 때 같은 데이터가 보이는가? |
| **보호 (Protect)** | 내 데이터를 아무나 못 보게 | 로그아웃하거나 다른 계정으로 들어가도 내 것이 보이는가? |

> **위 3개 중 하나라도 "아니오"** 라면 백엔드가 붙지 않았거나, 붙었어도 잘못 붙어 있습니다. 이번 주에 고쳐야 합니다.

### 4. API = 주문서

5주차의 레스토랑 비유를 떠올리세요. 프론트(홀)가 백엔드(주방)에 보내는 **주문서가 API**입니다.

```
프론트엔드                         백엔드 + DB
─────────────                     ─────────────
"매칭 목록 줘"    ───── GET ─────►  목록 조회
                                    │
"이 매칭 만들어"  ───── POST ────►  새 row 저장
                                    │
"이 매칭 수정해" ───── PATCH ────►  row 업데이트
                                    │
"이 매칭 지워"   ───── DELETE ───►  row 삭제
```

> **여러분 코드 안에 이미 이런 호출이 여러 개 들어 있습니다.** 직접 쓴 적이 없어도 AI가 써 놓았습니다. 그걸 찾고 이해하는 게 오늘의 출발점입니다.

### 5. CRUD — 이미 만든 기능에 이름표 붙이기

| 약자 | HTTP 메서드 | 일상 언어 | 밥메이트 예시 |
|-----|------------|---|-------------|
| **C**reate | POST | 새로 만들기 | "매칭 요청 등록" 버튼 |
| **R**ead | GET | 읽어오기 | 오늘의 매칭 목록 화면 |
| **U**pdate | PATCH / PUT | 수정 | 프로필 사진 변경 |
| **D**elete | DELETE | 삭제 | 내 요청 취소 |

#### 🧠 오늘의 첫 프롬프트 — "내 프로젝트 구조 설명해줘"

```
내 프로젝트 전체를 훑어본 다음, 아래 형식으로 정리해줘:

1. 사용 중인 기술 스택 (프론트/백엔드/DB)
2. 데이터베이스 테이블 목록과 각 필드
3. 이 프로젝트의 모든 기능을 CRUD 관점에서 표로 정리
   | 기능 이름 | CRUD | 관련 파일 | 관련 테이블 | 설명 |
4. 내가 놓치고 있거나 잘못 구현된 부분이 있으면 알려줘
```

> 이 프롬프트 하나면 여러분이 만든 것이 AI의 입으로 정리되어 돌아옵니다. **"내 코드를 내가 모른다"** 상태에서 벗어나는 첫 걸음입니다.

#### 🧠 "이 화면에서 뭐가 일어나는지 설명해줘"

```
[화면 스크린샷 또는 파일 경로]

이 화면에서 "[버튼 이름]" 을 눌렀을 때
- 어떤 API가 호출되는가?
- 어떤 테이블에 어떤 데이터가 저장/조회되는가?
- 화면은 어떻게 바뀌는가?
- 에러가 나면 어디에서 처리되는가?
를 단계별로 설명해줘.
```

> CRUD보다 더 복잡한 동작(예: "결제 후 재고 차감 + 이메일 발송")도 결국 **여러 CRUD의 조합**입니다. "이건 Create 1개 + Update 2개 + 알림 발송이네" 하고 쪼개는 눈을 기르세요.

### 6. 인증(AuthN) vs 인가(AuthZ) — 한 문장 구분

"보호(Protect)"는 **두 가지 질문**에 답하는 일입니다.

| 개념 | 질문 | 체크하는 것 | 예시 |
|------|------|-----------|------|
| **인증 (Authentication, AuthN)** | "너 누구야?" | 신분 확인 | 로그인, 비밀번호, 세션, JWT |
| **인가 (Authorization, AuthZ)** | "그걸 할 수 있어?" | 권한 확인 | "이 포스트를 삭제할 수 있는가?" |

**흔한 초보 실수:**

> ❌ "로그인했으면 뭐든 할 수 있다"
> ✅ "로그인은 신분증 확인일 뿐. 권한은 그 다음 단계."

#### 🧠 "내 프로젝트의 인증/인가 상태 점검해줘"

```
내 프로젝트를 열어서 다음을 점검해줘:

1. 인증(AuthN) — 로그인 시스템이 붙어 있는가? 어떤 방식인가?
   (Supabase Auth / Firebase Auth / 직접 구현 / 없음)
2. 인가(AuthZ) — 로그인한 사용자가 "아무거나" 할 수 있는가?
   권한 분기가 어디에도 없다면 경고해줘.
3. 보안 허점 — 평문 비밀번호, 하드코딩된 키, service_role 노출 등이
   있으면 전부 찾아서 리스트로 알려줘.
```

### 7. RBAC — 바이브 코딩의 "마법 키워드" 🪄

**RBAC = Role-Based Access Control (역할 기반 접근 제어)**

> **"이 단어 하나만 프롬프트에 던져도 AI가 만드는 결과물의 퀄리티가 완전히 달라집니다."**

#### 핵심 아이디어

사용자에게 **직접 권한을 주지 않고**, "역할"을 거쳐서 준다.

```
사용자 ──(user_roles)──► 역할 ──(role_permissions)──► 권한
김이화  ───────────►  editor  ────────────────►  post:create
                                                   post:read
                                                   post:update
                                                   post:delete
```

#### 왜 이렇게 하나? — "중복 없는 관리"

| 상황 | 직접 권한 부여 (나쁜 예) | RBAC (좋은 예) |
|------|----------------------|---------------|
| 관리자 10명에게 새 권한 추가 | 10명에게 각각 추가 | `role_permissions` 1줄 추가 → 10명에게 자동 전파 |
| 특정 사용자를 "에디터"로 강등 | 권한 수십 개 하나씩 제거 | `user_roles` 1줄만 변경 |
| 새 관리자 추가 | 권한 수십 개 복사 | `user_roles` 1줄 추가 |

> 이 **"한 줄만 바꾸면 전파"** 가 RBAC의 진짜 가치입니다. **여러분이 지금 `if user.is_admin`, `if user.is_vip` 같은 if 문을 난사하고 있다면 RBAC으로 갈아탈 때입니다.**

### 8. RBAC 테이블 설계 — 구조만 눈에 익혀두기

RBAC은 **5개 테이블**로 구성됩니다. 코드를 외울 필요는 없고, **다이어그램이 머릿속에 있으면** 프롬프트가 정확해집니다.

```
┌────────────┐       ┌────────────┐        ┌─────────────────┐
│   users    │       │   roles    │        │  permissions    │
├────────────┤       ├────────────┤        ├─────────────────┤
│ id         │       │ id         │        │ id              │
│ email      │       │ name       │        │ resource        │
│ password_  │       │ description│        │ action          │
│  hash      │       │            │        │                 │
└─────┬──────┘       └─────┬──────┘        └────────┬────────┘
      │                    │                        │
      │   ┌────────────────┴─────┐    ┌─────────────┴──────┐
      │   │    user_roles        │    │  role_permissions  │
      │   ├──────────────────────┤    ├────────────────────┤
      └──►│ user_id  │  role_id  │◄───┤ role_id │ perm_id  │
          └──────────────────────┘    └────────────────────┘
              (다대다 연결)                (다대다 연결)
```

> ⚠️ **절대 하지 말 것**: 사용자 테이블에 `is_admin boolean` 같은 플래그를 박는 것. 처음엔 편해 보이지만 역할이 2개만 돼도 무너집니다. 이 경고를 **프롬프트에 그대로 넣으면** AI가 제대로 된 테이블을 만들어줍니다.

#### 권한 명시 컨벤션 — `리소스:액션`

업계 표준입니다. OAuth 2.0 스코프(scope)에도 똑같이 씁니다.

```
post:create    post:read    post:update    post:delete
user:manage    comment:*    (와일드카드)
```

**CMS 4가지 역할 예시:**

| 역할 | 권한 | 한 줄 설명 |
|------|------|----------|
| **admin** | `post:*`, `user:*`, `comment:*` | 모든 것을 할 수 있음 |
| **editor** | `post:create/read/update/delete` (남의 글 포함) | 포스트 전권 |
| **author** | `post:create/read`, 본인 글만 `update/delete` | 내 글만 관리 |
| **viewer** | `post:read` | 읽기 전용 |

> AI에게 **"권한을 `resource:action` 형식으로 명시해줘"** 라고만 하면 이 컨벤션으로 결과가 나옵니다.

### 9. RBAC vs ABAC vs ReBAC — 프롬프트 용어 3종 세트

셋 다 `xAC = Access Control`의 약자입니다. 이름의 앞글자가 **"무엇을 기준으로 허용하는가"** 를 뜻합니다.

| 모델 | 풀네임 | 기준 | 예시 판단 | 언제 쓰나 |
|------|--------|------|---------|---------|
| **RBAC** | **R**ole-Based AC | 역할 | "editor이면 수정 가능" | **대부분의 MVP → 기본값** |
| **ABAC** | **A**ttribute-Based AC | 속성 | "09~18시 + 본인 부서 + 100만원 이하이면 승인" | 금융·의료·복잡 조건 |
| **ReBAC** | **Re**lationship-Based AC | 관계 | "내가 초대한 사람은 내 문서에 댓글 가능" | Google Docs, Notion, Figma |

> **결론**: MVP는 **RBAC으로 시작**. 알기만 해도 AI에게 정확한 지시가 가능해집니다. **"알고 있다"와 "모른다"의 차이는 프롬프트에서 드러납니다.**

### 10. UI 권한 ≠ 서버 권한 — 둘 다 체크해야 한다

> ❌ "에디터 버튼을 UI에서 숨겼으니까 뷰어는 못 누르겠지"

뷰어가 **개발자 도구를 열어서 API를 직접 호출**하면 삭제가 됩니다. UI 숨김은 "안내"일 뿐, **실제 차단은 서버에서 일어나야** 합니다.

**원칙**: **"UI 권한은 친절함, 서버 권한은 보안."** 둘 다 있어야 합니다.

#### 🧠 "내 권한 체크가 허술한지 점검해줘"

```
내 프로젝트에서 권한(authorization) 관련 코드를 전부 찾아서 다음을 해줘:

1. UI 에서만 권한 체크하고 서버에선 안 하는 엔드포인트가 있는지 찾기
2. `if user.role === 'admin'` 같은 하드코딩된 권한 체크가 있는지 찾기
3. 서버 사이드(Route Handler / API Route)에서
   - 로그인 여부만 확인하고 실제 권한은 확인 안 하는 곳
   - 401(인증 실패)과 403(권한 부족)을 구분하지 않는 곳
4. 각 문제마다 위험도(High/Medium/Low)와 개선 방법을 알려줘

리팩토링 코드까지 바로 제안해주면 더 좋아.
```

### 11. 백엔드 옵션 — 지금 뭘 쓰고 있는지부터 확인

5주차에 여러 도구를 소개했지만, 백엔드는 **3가지 패턴** 중 하나입니다. 먼저 "내가 지금 뭘 쓰고 있는지" 확인하세요.

| 패턴 | 누가 쓰나 | 장점 | 단점 |
|------|---------|------|------|
| **A. Supabase** (PostgreSQL) | Cursor, Bolt, Lovable 쓰는 사람 | 무료, SQL, 인증·RBAC 구현 쉬움 | 초기 연결 5분 필요 |
| **B. Firebase/Firestore** | Google AI Studio 쓰는 사람 | "Enable Firebase" 한 번이면 끝 | NoSQL, 복잡한 쿼리 어려움 |
| **C. 내장 DB** (Bolt/Replit) | Bolt.new, Replit 쓰는 사람 | 아무 설정 없음 | 이식 어려움 |

#### 🧠 "지금 내 백엔드 뭐 쓰고 있지?"

```
이 프로젝트가 현재 어떤 백엔드를 사용하고 있는지 알려줘.
- DB 종류와 연결 방식
- 환경변수 목록과 각각의 역할
- 인증 방식 (Supabase Auth / Firebase Auth / 직접 구현 / 없음)
- 만약 "로컬 state 만 쓰고 있다" 면 그것도 알려줘 — 그러면 배포 후 사라짐
```

### 12. Supabase 5분 세팅 (아직 백엔드 없는 사람용)

이미 백엔드가 붙어 있으면 이 섹션은 건너뛰세요. **처음 붙이는 사람만** 따라합니다.

1. [supabase.com](https://supabase.com) → **Start your project** → GitHub 로그인
2. **New project** → 이름/비밀번호/Region(Seoul) → 생성 (2~3분 대기)
3. 왼쪽 **SQL Editor** → **New query** → 테이블 SQL 붙여넣기 → **Run**
4. 왼쪽 **Project Settings → API** → 3개 값 복사
	- Project URL
	- anon public key
	- service_role key (🔒 서버 전용)
5. AI에게: **"Supabase를 이 프로젝트에 연결해줘. 환경변수는 .env.local에 넣고 .gitignore 확인해줘."**

> ⚠️ **`service_role`** 키를 `'use client'` 파일이나 GitHub에 절대 넣지 마세요. 내 DB 전체를 아무나 조작할 수 있게 됩니다.

### 13. 프롬프트 카탈로그 — 오늘 가장 많이 쓸 6가지

바이브 코딩에서 직접 코드를 쓸 일은 거의 없습니다. 대신 **이 6개 프롬프트 패턴**을 상황에 맞게 골라 쓰세요.

#### ① 구조 파악 — "내 프로젝트 설명해줘"

```
내 프로젝트 전체를 훑어본 다음, 아래 형식으로 정리해줘:
1. 사용 중인 기술 스택 (프론트/백엔드/DB)
2. 데이터베이스 테이블 목록과 각 필드
3. 모든 기능을 CRUD 관점에서 표로 정리
   | 기능 이름 | CRUD | 관련 파일 | 관련 테이블 |
4. 내가 놓치고 있거나 잘못 구현된 부분
```

#### ② 기능 추가 — "이런 기능 붙여줘"

```
[SPEC 또는 기능 설명]

이 기능을 내 프로젝트에 추가해줘.
- CRUD 중 어떤 동작인지 먼저 판단하고
- 기존 테이블/컴포넌트를 최대한 재사용
- 환경변수는 .env.local 사용
- 작업이 끝나면 어떤 파일이 바뀌었는지 요약
```

#### ③ RBAC 리팩토링 — "권한 시스템 제대로 붙여줘"

```
내 프로젝트에 RBAC(Role-Based Access Control)을 적용해줘.

현재 상태:
- 지금 로그인/권한 체크는 [있음/없음/부분적]
- 분리하고 싶은 역할: [예: admin, editor, user]

요구사항:
- 테이블: users, roles, permissions, user_roles, role_permissions
  (다대다 연결 2개 포함)
- 권한은 `resource:action` 형식 (예: post:create)
- 서버 사이드에서 권한 체크 미들웨어 구현
- UI에서도 역할에 따라 버튼/메뉴 숨김
- 기존 `if user.is_admin` 같은 하드코딩된 체크는 전부 제거
- 401(AuthN 실패)과 403(AuthZ 실패)을 구분해줘

절대 하지 말 것:
- 사용자 테이블에 is_admin 같은 플래그 추가
- 평문 비밀번호 저장
```

#### ④ 보안 점검 — "내 코드 털어봐"

```
내 프로젝트를 보안 관점에서 점검해줘. 특히 다음을 확인:

1. 하드코딩된 API 키나 비밀번호가 있는지
2. `.env.local` 이 `.gitignore` 에 포함되어 있는지
3. `service_role` 키가 클라이언트 코드(`'use client'`)에 쓰였는지
4. 서버 권한 체크 없이 클라이언트에서만 막는 엔드포인트
5. SQL Injection / XSS 에 취약한 부분
6. CORS, CSRF 설정
7. Supabase RLS 정책이 설정되어 있는지

각 항목마다 위험도(High/Medium/Low)와 구체적 개선 방법을 알려줘.
```

#### ⑤ 리팩토링 — "이 부분 더 깔끔하게"

```
[파일 경로 또는 함수 이름]

이 부분을 리팩토링해줘. 조건:
- 동작은 절대 바꾸지 말 것
- 중복 제거 + 가독성 향상
- 변경 이유를 bullet로 설명
- 변경 전/후 핵심 차이를 diff 형식으로 보여줘
```

#### ⑥ 배포 준비 — "내 프로젝트 배포 가능한가"

```
내 프로젝트를 Vercel에 배포하기 전에 점검해줘:

1. `npm run build` 가 로컬에서 통과하는지 (안 되면 에러 원인과 수정)
2. 필요한 환경변수 목록 + 각각의 용도
3. `.env.local` 이 git에 올라가지 않았는지 확인
4. README 에 배포 안내가 있는지 (없으면 간단히 작성)
5. 빌드 경고(warning)가 있다면 우선순위별 리스트
```

> 💡 이 6개 프롬프트를 **복사해서 개인 메모장에 저장**해두세요. 7주차 IR Day 전까지 반복해서 쓰게 됩니다.

### 14. 첫 MVP에서 RBAC 어디까지 갈까?

로그인·권한은 개발 시간을 2~3배 늘립니다. 7주차 IR Day까지 D-7인 지금, **세 가지 옵션** 중에 고르세요.

## Part 3: 배포 준비 — 지금부터 뭘 해야 하는가 (20분)

### 15. 배포가 뭔가요?

**배포(Deploy)** = 내 컴퓨터에서만 돌던 코드를 **인터넷 주소(URL)**로 만드는 일.

```
배포 전:
내 노트북 http://localhost:3000   ← 나만 볼 수 있음

배포 후:
https://bobmate.vercel.app        ← 친구, 교수님, 투자자 전부 볼 수 있음
```

### 16. 배포 파이프라인 — 머릿속에 넣으세요

```
내 코드                    GitHub                 Vercel
(로컬)                   (저장소)               (인터넷)
   │                        │                     │
   │  git push              │   자동 감지           │
   ├──────────────────────►│────────────────────►│
   │                        │                     │
   │                        │                     ▼
   │                        │              build & deploy
   │                        │                     │
   │                        │                     ▼
   │                        │          https://xxx.vercel.app
```

**핵심**: 한 번 연결해두면, 앞으로 `git push` 할 때마다 **자동으로 새 버전이 배포**됩니다.

### 17. 플랫폼 비교 — 뭘 고를까

| 플랫폼 | 장점 | 단점 | 추천 |
|-------|------|------|------|
| **Vercel** | Next.js 최적, 무료, 가장 쉬움 | Next.js 외엔 덜 편함 | **기본값** |
| **Netlify** | Vercel 동급, 팀 기능 좋음 | Next.js에선 Vercel보다 약간 불리 | 팀 작업 |
| **Firebase Hosting** | AI Studio 연동 한 번에 | 직접 설정 시 복잡 | AI Studio 사용자 |

### 18. 배포 전 체크리스트

- [ ] `npm run dev` 가 로컬에서 에러 없이 돌아감
- [ ] **`npm run build`** 가 에러 없이 통과됨 ← **이게 안 되면 배포도 100% 실패**
- [ ] `.env.local` 파일이 `.gitignore` 에 들어 있음
- [ ] API 키가 소스 코드에 하드코딩되어 있지 않음
- [ ] `service_role` 키가 클라이언트 코드에 없음

### 19. 환경변수 — 오늘 헷갈리면 안 되는 개념

환경변수는 **"코드에는 넣지 말고, 플랫폼 설정에만 넣는 비밀번호 메모장"**입니다.

| 이름 규칙 | 의미 | 예시 |
|----------|------|------|
| `NEXT_PUBLIC_*` | **브라우저에서도 읽힘** | `NEXT_PUBLIC_SUPABASE_URL` |
| 접두사 없음 | **서버에서만 읽힘 (비밀)** | `SUPABASE_SERVICE_ROLE_KEY` |

> ⚠️ **원칙**: 민감한 키에는 **절대 `NEXT_PUBLIC_` 붙이지 마세요.** Part 2.10의 "UI 권한 ≠ 서버 권한" 과 같은 원칙입니다.

전체 단계별 가이드는 **[스코입2026 — 서비스 배포 가이드](https://www.notion.so/33c68bb8a6608000b8dffec6e1d57380)** (Notion)에 있습니다. 오늘은 이 순서만 외워두세요:

### 21. 자주 터지는 배포 에러 TOP 5

| 에러 | 진짜 원인 | 해결 |
|------|---------|------|
| `Module not found: Can't resolve 'xxx'` | `npm install` 빠뜨리고 커밋 | `npm install xxx` → 커밋 → 푸시 |
| `Environment variable X is not defined` | Vercel에 env 안 넣음 or 오타 | Vercel → Settings → Env → **Redeploy** |
| `Row Level Security` 에러 | Supabase 테이블에 정책 없음 | Supabase → Authentication → Policies |
| 배포는 되는데 화면이 하얗게 뜸 | `NEXT_PUBLIC_` 안 붙임 | 변수명 수정 → Redeploy |
| `localhost:3000` 로 리다이렉트 | `NEXT_PUBLIC_BASE_URL` 을 배포 URL로 안 바꿈 | Vercel env 업데이트 → Redeploy |

> 에러 해결도 **에러 메시지 전체 + 내가 하려던 것 한 줄**을 AI에게 던지는 게 가장 빠릅니다.

## Part 5: 마무리 (10분)

### 27. 오늘 배운 것

- **내가 이미 만든 것에 이름표 붙이기**: CRUD, API, 인증, 인가
- **CRUD**: 모든 기능은 Create / Read / Update / Delete 네 가지로 쪼개진다
- **인증(AuthN) vs 인가(AuthZ)**: "너 누구야?" ≠ "그걸 할 수 있어?"
- **RBAC**: 사용자 → 역할 → 권한. 중복 없이 권한 관리
- **권한 컨벤션**: `리소스:액션` (post:create, user:manage)
- **UI 권한 ≠ 서버 권한**: UI는 친절함, 서버는 보안
- **프롬프트 카탈로그 6가지**: 구조 파악 / 기능 추가 / RBAC 리팩토링 / 보안 점검 / 리팩토링 / 배포 준비
- **배포 파이프라인**: 코드 → GitHub → Vercel → URL

> **오늘의 진짜 수확**: **"AI에게 내 프로젝트를 시켜본 경험"** 입니다. 오늘 쓴 프롬프트들을 저장해두세요. 앞으로 계속 씁니다.

## 부록 A: AI가 만든 코드 "읽는 법" — CRUD 빠른 레퍼런스

> **이 코드를 직접 쓸 일은 거의 없습니다.** AI가 여러분 프로젝트에 이미 비슷한 코드를 써 놨을 가능성이 높고, 여러분이 할 일은 그걸 **"읽어서 이해"** 하는 것입니다. 이해가 되면 "여기 이 부분 이렇게 바꿔줘" 같은 정확한 프롬프트가 가능해집니다.

### Supabase (JavaScript)

```javascript
// Create — POST /api/match-requests 뒤에서 이런 일이 일어남
const { data, error } = await supabase
  .from('match_requests')
  .insert({ user_name: '김이화', preferred_area: '정문', time_slot: '12:00' })

// Read — 목록 화면이 떴을 때 이런 일이 일어남
const { data, error } = await supabase
  .from('match_requests')
  .select('*')
  .order('created_at', { ascending: false })
  .limit(10)

// Update — "수정" 버튼을 눌렀을 때 이런 일이 일어남
const { data, error } = await supabase
  .from('match_requests')
  .update({ status: 'matched' })
  .eq('id', 42)

// Delete — "삭제" 버튼을 눌렀을 때 이런 일이 일어남
const { error } = await supabase
  .from('match_requests')
  .delete()
  .eq('id', 42)
```

### Firebase Firestore (JavaScript)

```javascript
// Create
await addDoc(collection(db, 'match_requests'), {
  user_name: '김이화',
  preferred_area: '정문',
  created_at: serverTimestamp()
})

// Read
const snap = await getDocs(
  query(collection(db, 'match_requests'), orderBy('created_at', 'desc'), limit(10))
)

// Update
await updateDoc(doc(db, 'match_requests', '42'), { status: 'matched' })

// Delete
await deleteDoc(doc(db, 'match_requests', '42'))
```

> 위 스니펫을 **AI 출력물과 비교해보는 연습**을 하세요. "아 이게 Create구나", "아 여기서 필터링하는구나" — 이 감각이 7주차 IR Day의 Q&A 시간에 여러분을 다르게 만듭니다.

## 부록 B: RBAC 권한 체크 코드 스니펫 (참고용)

> 이것도 직접 쓸 필요는 없습니다. **AI에게 "이런 구조로 만들어줘"** 할 때 참고 자료로 쓰세요.

### 서버 사이드 권한 체크 (Next.js Route Handler)

```javascript
// app/api/posts/[id]/route.ts
import { createServerClient } from '@supabase/ssr'

export async function DELETE(req, { params }) {
  const supabase = createServerClient(/* ... */)

  // 1. 인증(AuthN): 로그인한 사람인가?
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return Response.json({ error: 'Unauthorized' }, { status: 401 })

  // 2. 인가(AuthZ): 이 포스트를 삭제할 권한이 있는가?
  const { data: hasPermission } = await supabase
    .rpc('user_has_permission', {
      uid: user.id,
      resource: 'post',
      action: 'delete',
    })

  if (!hasPermission) {
    return Response.json({ error: 'Forbidden' }, { status: 403 })
  }

  // 3. 권한 통과 → 실제 삭제
  await supabase.from('posts').delete().eq('id', params.id)
  return Response.json({ ok: true })
}
```

> **401 vs 403** — 401(Unauthorized)은 "너 누군지 모르겠어"(AuthN 실패), 403(Forbidden)은 "누군지는 아는데 권한 없어"(AuthZ 실패). 이걸 구분 안 하는 코드는 AI에게 "401과 403을 구분해줘" 한 마디로 고칠 수 있습니다.

### 클라이언트 사이드 UI 숨김 (React)

```tsx
function PostCard({ post, userRole }) {
  const canEdit = userRole === 'admin' || userRole === 'editor'
  const canDelete = userRole === 'admin'

  return (
    <div>
      <h3>{post.title}</h3>
      {canEdit && <button>수정</button>}
      {canDelete && <button>삭제</button>}
    </div>
  )
}
```

> UI 숨김은 **편의**일 뿐입니다. 실제 차단은 서버(위 코드)에서 일어나야 합니다.
