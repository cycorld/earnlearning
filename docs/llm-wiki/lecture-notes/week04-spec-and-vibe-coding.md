### 4. SPEC이란 무엇인가? (10분)

#### 기술 명세서(SPEC)의 역할

PRD가 **"무엇을 만들지"**를 정의했다면, SPEC은 **"어떻게 만들지"**를 정의합니다.

| PRD (지난 주) | SPEC (오늘) |
|---|---|
| 사용자 스토리 | 유저 플로우 시나리오 |
| 화면 목록 (Must/Should/Nice-to-have) | UI 와이어프레임 |
| 기능 상세 (입력/처리/출력/예외) | API 설계 (엔드포인트/요청/응답) |
| — | 데이터 모델 (테이블/필드/관계) |
| — | 기술 스택 선정 |
| — | 디자인 스펙 (컬러/폰트/간격) |

#### SPEC의 5가지 핵심 구성요소

1. **유저 플로우** — 사용자가 서비스를 이용하는 전체 시나리오
2. **UI 와이어프레임** — 각 화면의 레이아웃과 구성 요소
3. **디자인 스펙** — 컬러, 폰트, 간격, 컴포넌트 규격
4. **데이터 모델** — 테이블 구조, 필드, 타입, 관계
5. **API 설계** — 엔드포인트, 요청/응답 형식, 에러 처리

#### 왜 SPEC이 중요한가?

> SPEC = AI에게 줄 프롬프트의 원재료

- SPEC이 구체적일수록 → AI가 생성하는 코드 품질이 높아짐
- SPEC이 모호하면 → AI가 임의로 판단 → 원하는 결과물이 안 나옴
- "자연어로 코드를 작성한다" = "좋은 SPEC을 작성한다"

---

### 5. AI를 활용한 SPEC 자동 생성 (10분)

#### 프롬프트 한 줄이면 됩니다

실제로 강사의 EarnLearning 프로젝트에서 사용한 프롬프트:

```
PRD를 토대로 SPEC 문서를 작성해서 개발 준비해줘.
```

> 이 한 줄로 기술 스택, 데이터 모델, API 설계, 인증 방식까지 포함된 SPEC이 자동 생성되었습니다.

#### 좀 더 구체적인 프롬프트 (권장)

```
아래 PRD를 기반으로 기술 SPEC 문서를 작성해줘.

[PRD 전체 내용 붙여넣기]

SPEC에 다음을 포함해줘:
1. 기술 스택 (프론트엔드, 백엔드, DB, 배포)
2. 유저 플로우 시나리오 (Must 기능 기준)
3. UI 와이어프레임 (ASCII 형식)
4. 디자인 스펙 (컬러 팔레트, 폰트, 간격)
5. 데이터 모델 (테이블, 필드, 타입)
6. API 설계 (엔드포인트, 요청/응답, 에러)

조건:
- 기술 스택은 Next.js + Tailwind CSS + Supabase + Vercel
- 1인 개발자가 2주 내 구현 가능한 범위
- Must 기능만 상세하게, Should/Nice-to-have는 "향후 추가" 표시
```

#### SPEC 생성 후 검토 포인트

- [ ] PRD의 Must 기능이 모두 SPEC에 반영되었는가?
- [ ] 데이터 모델에 빠진 필드가 없는가?
- [ ] API 엔드포인트가 기능별로 정의되었는가?
- [ ] 기술 스택이 현실적인가? (배운 도구 기준)

### 6. SPEC 예시: 밥메이트 (15분)

#### 6-1. 유저 플로우

```
[앱 접속]
    ↓
[홈 화면] — 현재 위치 자동 표시
    ↓
[밥 먹으러 갈래요!] 버튼 탭
    ↓
[식당 영역 선택] — 정문/후문/학교 앞
    ↓
[매칭 대기] — 30초 타이머
    ↓ (매칭 성공)          ↓ (매칭 실패)
[매칭 결과 화면]       [다시 시도 안내]
  - 상대 프로필 카드
  - 추천 식당 + 지도
    ↓
[만남 완료]
```

#### 6-3. 디자인 스펙

| 항목 | 값 |
|------|-----|
| Primary Color | #00462A (이화 그린) |
| Accent Color | #4CAF50 |
| Background | #FFFFFF |
| Card Background | #F5F5F5 |
| Font | Pretendard |
| Title | 24px Bold |
| Body | 14px Regular |
| Caption | 12px Light |
| Border Radius | 8px |
| Spacing Grid | 16px |
| CTA 버튼 높이 | 48px |
| 하단 탭바 높이 | 56px |

#### 6-4. 데이터 모델

**users 테이블**

| 필드 | 타입 | 설명 |
|------|------|------|
| id | uuid | 사용자 고유 ID (PK) |
| email | text | 학교 이메일 |
| name | text | 이름 |
| department | text | 학과 |
| grade | int | 학년 |
| interests | text[] | 관심사 배열 |
| created_at | timestamp | 가입일 |

**match_requests 테이블**

| 필드 | 타입 | 설명 |
|------|------|------|
| id | uuid | 요청 ID (PK) |
| user_id | uuid | 요청자 (FK → users) |
| preferred_area | text | 희망 식당 영역 |
| time_slot | timestamp | 식사 희망 시간 |
| status | text | pending / matched / expired |
| matched_with | uuid | 매칭 상대 (FK → users) |
| created_at | timestamp | 요청 생성 시간 |

#### 6-5. API 설계

| Method | Endpoint | 설명 | 요청 | 응답 |
|--------|----------|------|------|------|
| POST | /api/match | 매칭 요청 | `{ preferred_area, time_slot }` | `{ request_id, status }` |
| GET | /api/match/:id | 매칭 결과 조회 | — | `{ matched_user, restaurant }` |
| PATCH | /api/match/:id | 매칭 수락/거절 | `{ action: "accept" }` | `{ status }` |
| GET | /api/profile | 내 프로필 조회 | — | `{ user }` |
| PUT | /api/profile | 프로필 수정 | `{ department, interests }` | `{ user }` |

## Part 3: 바이브 코딩 시작 (60분)

### 7. 바이브 코딩 도구 선택 가이드 (15분)

바이브 코딩 도구는 하나가 아닙니다. **자신의 수준과 프로젝트에 맞는 도구**를 골라보세요.

#### 코딩 경험별 추천 경로

| 경험 수준 | 추천 도구 | 특징 |
|----------|----------|------|
| **코딩 처음** | Lovable 또는 Bolt.new | 자연어 입력 → 완성된 앱 생성, 설치 불필요 |
| **조금 해봄** | Cursor + Claude | 코드 직접 제어 가능, 자유도 높음 |
| **좀 할 줄 앎** | Claude Code (CLI) | 터미널에서 AI와 페어 프로그래밍 |

> 어떤 도구를 쓰든 **SPEC이 좋으면 결과물이 좋다**는 원칙은 동일합니다.

#### 도구별 상세 비교

##### A. Lovable (구 GPT Engineer) — 가장 쉬운 시작

| 항목 | 내용 |
|------|------|
| 사이트 | [lovable.dev](https://lovable.dev) |
| 방식 | 브라우저에서 자연어로 앱 생성 |
| 장점 | 설치 없음, 즉시 시작, 예쁜 기본 디자인 |
| 단점 | 세부 제어 어려움, 무료 크레딧 제한 |
| 적합 | 빠르게 프로토타입 만들고 싶을 때 |

**시작 방법:**
1. lovable.dev 접속 → 회원가입
2. "New Project" → SPEC 내용 붙여넣기
3. AI가 전체 앱 생성 → 실시간 미리보기
4. 수정 요청: "버튼 색상을 초록색으로 바꿔줘"

##### B. Bolt.new — 브라우저에서 풀스택

| 항목 | 내용 |
|------|------|
| 사이트 | [bolt.new](https://bolt.new) |
| 방식 | 브라우저 IDE + AI 코드 생성 |
| 장점 | 설치 없음, 코드 직접 수정 가능, Supabase 연동 지원 |
| 단점 | 무료 크레딧 제한, 복잡한 로직은 한계 |
| 적합 | 코드를 보면서 배우고 싶을 때 |

**시작 방법:**
1. bolt.new 접속 → 회원가입
2. 프롬프트에 SPEC 붙여넣기
3. 코드 + 미리보기 동시 확인
4. 코드를 직접 수정하거나 AI에게 추가 요청

##### C. v0.dev — UI 컴포넌트 생성 특화

| 항목 | 내용 |
|------|------|
| 사이트 | [v0.dev](https://v0.dev) |
| 방식 | 자연어 → React + Tailwind UI 컴포넌트 생성 |
| 장점 | UI 퀄리티가 매우 높음, shadcn/ui 기반 |
| 단점 | UI만 생성 (백엔드/DB 별도), 코드 복사 후 프로젝트에 통합 필요 |
| 적합 | 예쁜 화면을 먼저 만들고 싶을 때, Cursor와 병행 |

**시작 방법:**
1. v0.dev 접속
2. "밥메이트 홈 화면: 위치 표시, CTA 버튼, 식당 선택 탭" 입력
3. 생성된 컴포넌트 코드를 Cursor 프로젝트에 복사

##### D. Cursor + Claude — 자유도 최고

| 항목 | 내용 |
|------|------|
| 사이트 | [cursor.com](https://cursor.com) |
| 방식 | VS Code 기반 에디터 + AI 내장 |
| 장점 | 코드 완전 제어, 어떤 기술 스택이든 사용 가능, 파일 구조 직접 관리 |
| 단점 | 설치 필요, 터미널 기본 지식 필요 |
| 적합 | 코드를 직접 다루고 싶을 때 |

**Cursor에서 AI에게 말하는 법:**

| 방법 | 단축키 | 용도 |
|------|--------|------|
| **Chat** | Cmd+L | 대화하며 코드 생성/수정 |
| **Inline Edit** | Cmd+K | 선택한 코드를 직접 수정 |
| **Composer** | Cmd+I | 여러 파일 동시 생성/수정 |

##### E. Replit — 협업 + 즉시 배포

| 항목 | 내용 |
|------|------|
| 사이트 | [replit.com](https://replit.com) |
| 방식 | 브라우저 IDE + AI Agent + 즉시 배포 |
| 장점 | 설치 없음, URL 공유로 즉시 시연, AI Agent가 자동 코딩 |
| 단점 | 무료 플랜 제한, 속도 느릴 수 있음 |
| 적합 | 빠르게 배포해서 다른 사람에게 보여주고 싶을 때 |

##### F. Claude Code (CLI) — 파워 유저용

| 항목 | 내용 |
|------|------|
| 설치 | `npm install -g @anthropic-ai/claude-code` |
| 방식 | 터미널에서 AI와 대화하며 코딩 |
| 장점 | 가장 강력한 코드 생성, 파일 생성/수정/실행 자동화 |
| 단점 | 터미널 경험 필요, API 키 필요 |
| 적합 | 코딩 경험이 있고, 대규모 프로젝트를 빠르게 만들고 싶을 때 |

#### 도구 조합 추천

도구는 하나만 써야 하는 게 아닙니다. **조합해서 쓰면 더 강력합니다.**

| 조합 | 용도 |
|------|------|
| **v0.dev → Cursor** | v0에서 UI 생성 → Cursor에서 백엔드 연결 |
| **Lovable → 내보내기 → Cursor** | Lovable로 프로토타입 → 코드 내보내서 Cursor에서 확장 |
| **Bolt.new → Vercel 배포** | Bolt에서 개발 → GitHub 연동 → Vercel 자동 배포 |
| **ChatGPT/Claude → Cursor** | AI에게 설계 상담 → Cursor에서 코딩 |

#### 백엔드/DB 선택지

| 도구 | 특징 | 추천 상황 |
|------|------|----------|
| **Supabase** | PostgreSQL + Auth + API 자동 생성 | DB가 필요한 대부분의 프로젝트 |
| **Firebase** | NoSQL + Auth + 호스팅 | 실시간 채팅, 간단한 데이터 저장 |
| **없음 (프론트만)** | LocalStorage 또는 목업 데이터 | MVP 검증 단계, 최대한 빠르게 |

#### 배포 선택지

| 도구 | 특징 | 추천 상황 |
|------|------|----------|
| **Vercel** | Next.js 최적화, GitHub 연동 자동 배포 | Cursor + Next.js 조합일 때 |
| **Netlify** | 정적 사이트 + 서버리스 | 간단한 웹사이트 |
| **Lovable/Bolt 내장 배포** | 클릭 한 번으로 배포 | 해당 도구로 만들었을 때 |
| **Replit** | 만든 그대로 즉시 공유 URL | Replit으로 만들었을 때 |

### 8. 바이브 코딩 실전 (45분)

#### 도구별 시작 프롬프트

**모든 도구 공통**: SPEC 문서를 통째로 AI에게 전달합니다.

##### Lovable / Bolt.new / Replit 사용자

```
아래 SPEC을 기반으로 웹 서비스를 만들어줘.

[SPEC 전체 내용 붙여넣기]

조건:
- Must 화면부터 먼저 구현
- 디자인 스펙의 색상과 폰트 적용
- 모바일 반응형
```

##### Cursor 사용자

Composer(Cmd+I)에서:
```
아래 SPEC 문서를 기반으로 Next.js 프로젝트를 세팅해줘.

[SPEC.md 내용 또는 @SPEC.md 태그]

조건:
- Next.js 14 + App Router
- Tailwind CSS
- Supabase 연동 (환경변수 .env.local)
- 기본 레이아웃 + 하단 탭바
- Must 화면의 빈 페이지 생성
```

##### v0.dev 사용자

```
밥메이트 앱의 홈 화면을 만들어줘.

- 상단: 서비스 로고 + 메뉴 아이콘
- 중앙: 현재 위치 표시 + "밥 먹으러 갈래요!" 큰 버튼
- 하단: 식당 영역 선택 칩 (정문/후문/학교앞) + 탭바
- 색상: Primary #00462A, Accent #4CAF50
- 모바일 사이즈 (375px 기준)
```

#### Step 2: 첫 화면에서 두 번째 화면으로 (20분)

첫 화면이 완성되면 다음을 요청하세요:

```
홈 화면에서 "밥 먹으러 갈래요" 버튼을 누르면
매칭 결과 화면으로 이동하도록 만들어줘.

매칭 결과 화면에는:
- 매칭된 상대 프로필 카드 (이름, 학과, 학년, 관심사)
- 추천 식당 정보
- 매칭 수락/취소 버튼
을 포함해줘.
```

#### Step 3: 개별 코칭 & 문제 해결 (15분)

- 각자 자신의 프로젝트 SPEC으로 코딩 시작
- 막히는 부분은 손 들고 질문
- 흔한 문제:

| 문제 | 해결 |
|------|------|
| "뭘 써야 할지 모르겠어요" | Lovable이나 Bolt.new로 시작해보세요 |
| Supabase 연결 안 됨 | 환경변수(.env.local) 확인 |
| 빌드 에러 | 에러 메시지를 AI에게 그대로 붙여넣기 |
| 디자인이 안 맞음 | SPEC 디자인 스펙 섹션을 AI에게 다시 참조시키기 |
| 무료 크레딧 소진 | 다른 도구로 전환 (Lovable → Bolt → Cursor) |

#### 바이브 코딩 팁

1. **SPEC을 항상 열어두기** — AI가 맥락을 잃으면 SPEC을 다시 참조시킴
2. **한 번에 하나씩** — "전체 다 만들어줘" 대신 "홈 화면만 만들어줘"
3. **에러는 그대로 복붙** — 에러 메시지를 AI에게 그대로 던지기
4. **자주 저장하고 확인** — 브라우저에서 결과 확인 후 다음 단계
5. **도구에 갇히지 말기** — 막히면 다른 도구로 같은 SPEC을 시도해보기
6. **스크린샷 활용** — "이 화면을 이렇게 바꿔줘" + 스크린샷 첨부
