### 3. 웹 vs 앱: 무엇이 다른가?

우리가 매일 쓰는 서비스는 크게 두 종류입니다.

| 구분 | 웹 (Web) | 앱 (Native App) |
|------|----------|-----------------|
| 접근 방법 | 브라우저에서 URL 입력 | 앱스토어에서 설치 |
| 설치 필요? | 없음 | 있음 (다운로드 필요) |
| 업데이트 | 서버만 바꾸면 즉시 반영 | 스토어 심사 + 사용자 업데이트 |
| 개발 비용 | 하나만 만들면 됨 | iOS + Android 따로 만들어야 함 |
| 기기 기능 | 제한적 (카메라, GPS 등 가능) | 전부 사용 가능 |
| 배포 속도 | 매우 빠름 (분 단위) | 느림 (심사 3~7일) |
| 대표 예시 | 당근마켓 웹, 네이버, Gmail | 카카오톡, 인스타그램 |

#### 왜 우리는 웹을 선택하는가?

1. **속도**: MVP를 빠르게 만들고 빠르게 배포할 수 있다
2. **비용**: 하나만 만들면 모든 기기에서 동작한다
3. **검증**: 사용자 반응을 빠르게 확인하고 수정할 수 있다
4. **스타트업의 원칙**: "일단 웹으로 검증하고, 되면 앱으로 확장"

> 당근마켓, 토스, 배달의민족 모두 웹 MVP로 시작했습니다.

#### PWA: 웹인데 앱 같은 것

**PWA(Progressive Web App)**는 웹사이트인데 앱처럼 동작하는 기술입니다.

- 홈 화면에 아이콘 추가 가능
- 오프라인에서도 동작
- 푸시 알림 지원
- 앱스토어 심사 없이 배포

> 우리가 사용하는 **언러닝(EarnLearning) LMS**도 PWA로 만들어져 있습니다.
> 홈 화면에 추가하면 앱처럼 사용할 수 있죠!

### 4. 웹 서비스의 3계층 구조

웹 서비스는 **레스토랑**에 비유할 수 있습니다.

```
┌─────────────────────────────────────────────────┐
│                  사용자 (손님)                     │
│                                                   │
│  "밥메이트에서 매칭 버튼을 누른다"                     │
└───────────────────┬─────────────────────────────┘
                    │ 요청 (주문)
                    ▼
┌─────────────────────────────────────────────────┐
│              프론트엔드 (홀 직원)                   │
│                                                   │
│  - 손님이 보는 화면 (메뉴판, 테이블)                  │
│  - 주문을 받아서 주방에 전달                          │
│  - 결과를 예쁘게 손님에게 보여줌                       │
│                                                   │
│  기술: HTML, CSS, JavaScript, React, Next.js       │
└───────────────────┬─────────────────────────────┘
                    │ API 호출 (주문서)
                    ▼
┌─────────────────────────────────────────────────┐
│              백엔드 (주방)                          │
│                                                   │
│  - 실제 로직을 처리 (요리)                           │
│  - 누가 어떤 요청을 했는지 검증                       │
│  - 데이터를 저장하고 불러옴                           │
│                                                   │
│  기술: Node.js, Python, Go, Java                   │
└───────────────────┬─────────────────────────────┘
                    │ 쿼리 (재료 꺼내기)
                    ▼
┌─────────────────────────────────────────────────┐
│              데이터베이스 (냉장고/창고)               │
│                                                   │
│  - 모든 데이터를 영구 저장                            │
│  - 사용자 정보, 게시글, 거래 내역 등                   │
│  - 요청에 따라 데이터를 찾아서 전달                     │
│                                                   │
│  기술: PostgreSQL, SQLite, MySQL, MongoDB          │
└─────────────────────────────────────────────────┘
```

#### 실제 예시: "밥메이트에서 매칭 버튼 누르기"

| 단계 | 계층 | 일어나는 일 |
|------|------|-----------|
| 1 | 프론트엔드 | 사용자가 "밥 먹으러 갈래요!" 버튼 클릭 |
| 2 | 프론트엔드 → 백엔드 | `POST /api/match` 요청 전송 |
| 3 | 백엔드 | 같은 시간, 같은 영역에 매칭 가능한 사람 검색 |
| 4 | 백엔드 → DB | "매칭 요청" 데이터를 저장하고, 가능한 상대를 조회 |
| 5 | DB → 백엔드 | 매칭 가능한 사용자 정보 반환 |
| 6 | 백엔드 → 프론트엔드 | 매칭 결과를 JSON으로 응답 |
| 7 | 프론트엔드 | 매칭 결과 화면을 사용자에게 보여줌 |

### 5. 각 계층 자세히 보기

#### 프론트엔드 (Frontend) — 사용자가 보는 모든 것

```
HTML  →  뼈대 (구조)        "버튼이 여기에 있다"
CSS   →  옷 (스타일)         "버튼을 초록색으로"
JS    →  두뇌 (동작)         "버튼 누르면 다음 화면으로"
```

**프론트엔드 프레임워크**: 복잡한 화면을 효율적으로 만들기 위한 도구

| 프레임워크 | 개발사 | 특징 | 사용 기업 |
|-----------|--------|------|----------|
| **React** | Meta(페이스북) | 가장 인기, 컴포넌트 기반 | 인스타그램, 넷플릭스, 에어비앤비 |
| **Next.js** | Vercel | React 기반 풀스택 프레임워크 | 틱톡, 나이키, 노션 |
| **Vue.js** | 에반 유 (오픈소스) | 쉬운 학습 곡선 | 알리바바, 그렙(프로그래머스) |
| **Svelte** | 리치 해리스 | 가볍고 빠름 | 뉴욕타임즈, Apple |
| **Angular** | Google | 대규모 앱에 강점 | Google, 삼성 |

#### 백엔드 (Backend) — 보이지 않는 두뇌

사용자에게 보이지 않지만, 모든 **비즈니스 로직**을 처리합니다.

- 회원가입 시 비밀번호 암호화
- 결제 처리
- 권한 확인 (이 사용자가 이 글을 삭제할 수 있는가?)
- 외부 API 연동 (지도, 결제, SMS 등)

**백엔드 언어 & 프레임워크**:

| 언어 | 프레임워크 | 특징 | 사용 기업 |
|------|-----------|------|----------|
| **JavaScript** | Node.js / Express | 프론트와 같은 언어 | 넷플릭스, 우버, 링크드인 |
| **Python** | Django / FastAPI | AI/ML과 시너지, 배우기 쉬움 | 인스타그램, 스포티파이 |
| **Go** | Echo / Gin | 빠른 성능, 동시 처리에 강함 | 구글, 당근마켓 |
| **Java** | Spring Boot | 대기업 표준 | 삼성, 네이버, 카카오 |
| **TypeScript** | Next.js API Routes | 풀스택 한 언어로 가능 | 많은 스타트업 |

#### 데이터베이스 (Database) — 모든 것을 기억하는 곳

데이터를 **테이블**(엑셀 시트와 비슷) 형태로 저장합니다.

**users 테이블 예시:**

| id | name | email | department | created_at |
|----|------|-------|-----------|------------|
| 1 | 김이화 | kim@ewha.ac.kr | 경영학과 | 2026-03-06 |
| 2 | 박이화 | park@ewha.ac.kr | 컴퓨터공학과 | 2026-03-06 |
| 3 | 이이화 | lee@ewha.ac.kr | 디자인학과 | 2026-03-07 |

**DB 종류:**

| DB | 유형 | 특징 | 적합한 경우 |
|----|------|------|-----------|
| **PostgreSQL** | 관계형(SQL) | 가장 강력한 오픈소스 DB | 대부분의 웹 서비스 |
| **SQLite** | 관계형(SQL) | 파일 하나로 동작, 설치 불필요 | 소규모 서비스, 프로토타입 |
| **MySQL** | 관계형(SQL) | 가장 오래되고 널리 사용 | 전통적인 웹 서비스 |
| **MongoDB** | 문서형(NoSQL) | 유연한 데이터 구조 | 빠르게 변하는 데이터 |
| **Firestore** | 문서형(NoSQL) | Google Firebase 연동 | 실시간 앱, 소규모 프로젝트 |

> **Supabase** = PostgreSQL + 인증 + API 자동 생성 (무료 티어)
> **Firebase** = Firestore + 인증 + 호스팅 (무료 티어)

### 6. 우리의 기술 스택

#### 메인 스택: Next.js 풀스택

```
┌─────────────────────────────────────┐
│           Next.js (풀스택)           │
│                                     │
│  ┌───────────┐   ┌──────────────┐  │
│  │ 프론트엔드  │   │   백엔드      │  │
│  │ React     │   │ API Routes   │  │
│  │ Tailwind  │   │ Server       │  │
│  │ CSS       │   │ Actions      │  │
│  └───────────┘   └──────┬───────┘  │
│                         │           │
│                  ┌──────┴───────┐  │
│                  │     DB       │  │
│                  │ PostgreSQL   │  │
│                  │ 또는 SQLite   │  │
│                  └──────────────┘  │
│                                     │
│         배포: Vercel (무료)          │
└─────────────────────────────────────┘
```

**왜 Next.js인가?**

| 이유 | 설명 |
|------|------|
| **풀스택** | 프론트 + 백엔드를 하나의 프로젝트에서 작성 |
| **한 언어** | JavaScript/TypeScript 하나만 알면 됨 |
| **AI 친화적** | 바이브 코딩 도구들이 Next.js를 가장 잘 지원 |
| **무료 배포** | Vercel에서 원클릭 무료 배포 |
| **취업 시장** | 스타트업에서 가장 많이 사용하는 프레임워크 |

#### DB 선택지

| 옵션 | 장점 | 단점 | 추천 상황 |
|------|------|------|----------|
| **Supabase (PostgreSQL)** | 강력한 DB + 인증 + API 자동 생성 | 설정이 약간 필요 | DB가 필요한 대부분의 MVP |
| **SQLite** | 설치 불필요, 파일 하나 | 동시 접속에 약함 | 초소형 프로젝트, 프로토타입 |
| **Firebase (Firestore)** | Google 생태계, 실시간 동기화 | SQL 아님, 복잡한 쿼리 어려움 | AI Studio로 개발 시 |
| **없음 (LocalStorage)** | 가장 빠른 시작 | 데이터가 브라우저에만 저장 | 데이터 저장이 필요 없는 MVP |

> **참고**: 우리 수업의 언러닝 LMS는 **Go + SQLite** 조합으로 만들어져 있습니다.
> SQLite만으로도 수십 명이 동시에 쓰는 서비스를 충분히 운영할 수 있다는 걸 증명하고 있죠!

#### 다른 가능한 스택 조합

자신의 프로젝트에 더 맞는 스택이 있을 수 있습니다:

| 조합 | 특징 | 적합한 프로젝트 |
|------|------|---------------|
| **Next.js + Supabase + Vercel** | 우리 수업 메인 스택, 가장 범용적 | 대부분의 웹 서비스 |
| **Next.js + Firebase** | Google 생태계, AI Studio와 시너지 | AI Studio로 개발하는 경우 |
| **Python (Flask/Django) + PostgreSQL** | AI/ML 기능이 핵심인 서비스 | AI 추천, 데이터 분석 서비스 |
| **HTML + CSS + JS (순수)** | 가장 단순, 학습용 | 정적 웹사이트, 랜딩 페이지 |

### 7. 이 구조를 바이브 코딩에서는 어떻게 쓰는가?

바이브 코딩에서는 이 구조를 **직접 코딩하지 않고 AI에게 설명**합니다.

```
❌ 직접 코딩:
const express = require('express');
const app = express();
app.get('/api/match', async (req, res) => {
  const result = await db.query('SELECT * FROM matches...');
  res.json(result);
});

✅ 바이브 코딩:
"매칭 버튼을 누르면 같은 식당 영역을 선택한 사람과 매칭해줘.
매칭 결과는 상대방 이름, 학과, 추천 식당을 보여줘.
데이터는 Supabase에 저장해줘."
```

하지만 **구조를 이해하면 AI에게 더 정확한 지시를 줄 수 있습니다:**

```
✅ 더 좋은 바이브 코딩:
"Next.js API Route로 매칭 API를 만들어줘.
- POST /api/match: 매칭 요청 생성 (preferred_area, time_slot 파라미터)
- GET /api/match/[id]: 매칭 결과 조회
- Supabase의 match_requests 테이블에 저장
- 같은 preferred_area + 30분 이내 time_slot인 pending 요청과 매칭"
```

> **구조를 몰라도 만들 수 있지만, 알면 10배 더 잘 만든다.**

### 8. 왜 도구 이야기를 다시 하는가?

4주차에 Lovable, Bolt, Cursor 등을 소개했지만, **2026년 3월 Google AI Studio가 풀스택 바이브 코딩 플랫폼으로 대폭 업그레이드**되었습니다.

핵심 원칙은 변하지 않습니다:

> **도구에 갇히지 말 것. SPEC이 좋으면 어떤 도구에서든 결과가 좋다.**

### 9. 무료 도구 비교표 (2026년 4월 기준)

| 도구 | 무료 한도 | 배포 | 백엔드/DB | 난이도 | 추천 |
|------|----------|------|----------|--------|------|
| **Google AI Studio** | Gemini API 프리 티어 (넉넉) | Firebase 호스팅 | Firestore + Auth 자동 세팅 | 매우 쉬움 | ★★★ |
| **Bolt.new** | 일 300K 토큰 / 월 100만 토큰 | 무료 호스팅 내장 | 무제한 DB | 쉬움 | ★★★ |
| **Replit** | Agent 크레딧 제한, 월 1,200분 | 무료 1개 앱 | 내장 DB | 매우 쉬움 | ★★☆ |
| **Lovable** | 일 5크레딧 (~3회 상호작용) | 무료 배포 | Supabase 연동 | 매우 쉬움 | ★★☆ |

## Part 4: 효과적인 프롬프트 패턴 (20분)

---

### 12. 바이브 코딩 = 프롬프트 엔지니어링

지난 주에 SPEC을 통째로 넣고 코드를 생성했습니다.
이번 주는 **AI와 더 효율적으로 소통하는 패턴**을 배웁니다.

#### 패턴 1: 점진적 빌드 (Incremental Build)

한 번에 모든 것을 요청하지 마세요. **한 화면씩, 한 기능씩.**

```
❌ 나쁜 예: "전체 앱을 만들어줘"
✅ 좋은 예: "홈 화면만 먼저 만들어줘"
```

**점진적 빌드 순서:**
```
1단계: 홈 화면 레이아웃 → 확인
2단계: 두 번째 화면 → 확인
3단계: 화면 간 네비게이션 연결 → 확인
4단계: 데이터 연동 (DB) → 확인
5단계: 인증 (로그인) → 확인
```

#### 패턴 2: 구체적 지시 (Specific Instructions)

AI는 구체적인 지시를 줄수록 정확한 결과를 냅니다.

```
❌ 나쁜 예:
"이 화면을 예쁘게 만들어줘"

✅ 좋은 예:
"이 화면을 수정해줘:
- 배경색: #F5F5F5
- 카드 모서리: 둥글게 (8px)
- 버튼: Primary Color #00462A, 텍스트 흰색
- 폰트: 제목 24px Bold, 본문 14px Regular
- 카드 간 간격: 16px"
```

#### 패턴 3: 에러 해결 (Error Resolution)

에러가 나면 **에러 메시지를 그대로 AI에게 복붙**하세요.

```
✅ 좋은 예:
"이런 에러가 나는데 고쳐줘:

Error: Cannot read properties of undefined (reading 'map')
  at UserList (src/components/UserList.tsx:15:23)
"
```

```
✅ 더 좋은 예:
"이런 에러가 나는데 고쳐줘:

[에러 메시지]

내가 하려던 것: 사용자 목록을 불러와서 카드로 보여주기
현재 상태: API 호출은 성공하는데 화면 렌더링에서 에러남"
```

#### 패턴 4: 스크린샷 + 텍스트 (Visual Reference)

"이 화면을 이렇게 바꿔줘" → 스크린샷 첨부

```
✅ 좋은 예:
"현재 화면 [스크린샷 첨부]
이 화면에서 다음을 수정해줘:
1. 상단 헤더를 초록색으로
2. 버튼을 화면 하단에 고정
3. 카드 사이에 구분선 추가"
```

> AI Studio, Bolt, Lovable, Cursor 모두 이미지 첨부를 지원합니다.

#### 패턴 5: 역할 부여 (Role Assignment)

```
✅ 좋은 예:
"너는 이화여대 학생들을 위한 밥 매칭 서비스를 만드는 시니어 프론트엔드 개발자야.
아래 SPEC을 기반으로 홈 화면을 구현해줘.
모바일 퍼스트로, 깔끔하고 심플한 디자인이어야 해."
```

#### 패턴 6: 되돌리기 & 분기 (Undo & Branch)

AI가 만든 결과가 마음에 안 들면?

| 도구 | 되돌리기 방법 |
|------|-------------|
| **AI Studio** | 이전 버전으로 롤백 |
| **Bolt.new** | 히스토리에서 이전 상태 복원 |
| **Lovable** | 변경 이력에서 되돌리기 |
| **Cursor** | Cmd+Z 또는 Git으로 복원 |

> **팁**: 큰 변경을 요청하기 전에 현재 상태가 잘 작동하는지 확인하고, 작동하면 "여기까지 저장"하세요.

- 만든 화면: 홈 화면 + 매칭 결과 화면
- 프롬프트 횟수: 약 10회
- 소요 시간: 50분

[스크린샷 첨부]

#5주차 #바이브코딩 #밥메이트

**보상:**
- 좋아요를 가장 많이 받은 게시글 → 보너스 자금 지급 (유동성 공급)
- 모든 참여자에게 과제 보상 지급

---

### B-3. 외주 마켓 활용하기

서로 도울 수 있는 외주를 주고받아 보세요!

**등록 가능한 외주 예시:**

| 외주 내용 | 예산 예시 | 설명 |
|----------|----------|------|
| 로고 디자인 | 50만~200만원 | AI로 로고 만들어주기 |
| UI 피드백 | 30만~100만원 | 내 서비스 화면 리뷰해주기 |
| 프롬프트 작성 | 50만~150만원 | SPEC 기반 프롬프트 대신 작성 |
| 데이터 입력 | 20만~50만원 | 테스트 데이터 만들기 |
| 사용자 테스트 | 30만~100만원 | 내 MVP 써보고 피드백 주기 |

> 에스크로 시스템으로 안전하게 거래됩니다.
> 의뢰자가 작업물을 승인해야 돈이 지급됩니다.

### B-4. 피드에서 서로 돕기

**#자유** 채널을 적극 활용하세요!

- 바이브 코딩 중 막힌 점 공유 → 다른 학생이 댓글로 도움
- 유용한 프롬프트 공유 → "이 프롬프트가 잘 먹혔어요"
- 에러 해결 공유 → "이 에러는 이렇게 해결했어요"

> 댓글로 도움을 주면 **참여도 점수**에 반영됩니다!

### B-5. 투자 라운드 예고 (7주차 이후)

7주차에 첫 번째 MVP가 완성되면 **투자 라운드**를 열 수 있습니다.

- 다른 학생의 회사에 투자 → 주주가 됨
- 배당금 수령 가능
- 기업가치가 올라가면 거래소에서 주식 매매 가능

**지금 할 수 있는 것:**
- 다른 학생의 프로젝트에 관심 갖기
- #투자라운지에서 아이디어 토론
- "이 서비스에 투자하고 싶다!" 댓글 남기기

### B-6. 자산 랭킹 확인

지갑 페이지에서 **자산 랭킹**을 확인할 수 있습니다.

**자산을 늘리는 방법:**
| 방법 | 예상 수익 |
|------|----------|
| 과제 보상 수령 | 과제별 다름 |
| 외주 수주 | 20만~200만원 |
| 투자 배당금 | 투자 비율에 따라 |
| 주식 거래 차익 | 매매에 따라 |

> **학기 말 최종 자산가치가 성적에 반영됩니다!**
>
> 총 자산 = 현금 잔고 + 보유 주식 시가 - 부채(대출)
