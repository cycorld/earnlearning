### 2. PR&FAQ란 무엇인가? (15분)

#### 아마존의 "역방향 일하기(Working Backwards)"

<img src="assets/working-backwards-cover.jpg" alt="순서 파괴 - Working Backwards 책 표지" width="200" />

> *순서 파괴* (콜린 브라이어 & 빌 카 저) — 아마존의 PR&FAQ 방법론을 소개한 책

아마존에서는 새로운 제품이나 서비스를 시작할 때, **코드를 한 줄도 쓰기 전에** PR&FAQ 문서를 먼저 작성합니다.

- **PR (Press Release)**: 제품이 완성된 후 발표할 가상의 보도자료
- **FAQ**: 고객과 내부 이해관계자가 물어볼 예상 질문과 답변

#### 왜 PR&FAQ인가?

| 일반적인 접근 | PR&FAQ 접근 |
|---|---|
| "이런 기능을 만들 수 있어" (기술 중심) | "고객이 이런 문제를 해결할 수 있어" (고객 중심) |
| 만들고 나서 시장 반응 확인 | 만들기 전에 가치 검증 |
| 기획서가 길고 복잡 | 1~2페이지로 핵심만 정리 |
| 팀원 간 이해도 차이 | 누구나 읽고 이해 가능 |

#### PR&FAQ가 스타트업에 중요한 이유

1. **생각 정리 도구**: 내 아이디어를 명확하게 구조화
2. **커뮤니케이션 도구**: 투자자, 팀원, 멘토에게 빠르게 설명
3. **검증 도구**: 글로 써보면 논리적 허점이 보임
4. **시간 절약**: 잘못된 방향으로 개발하는 것을 방지

### 3. PR&FAQ 작성법 (20분)

#### PR (보도자료) 구조

```
제목: [서비스명], [핵심 가치] 제공하는 [서비스 유형] 출시

부제목: [타겟 사용자]를 위한 [핵심 기능] 소개

도입부 (1~2문장):
- 언제, 누가, 무엇을 출시했는지
- 핵심 가치 한 줄 요약

문제 (1~2문장):
- 기존에 어떤 문제가 있었는지
- 왜 기존 솔루션이 불충분한지

솔루션 (2~3문장):
- 우리 서비스가 어떻게 문제를 해결하는지
- 핵심 기능 2~3가지

고객 인용구 (1~2문장):
- 가상의 사용자가 말하는 체험담
- 구체적인 숫자나 결과 포함

사용 방법 (2~3문장):
- 서비스를 어떻게 시작하는지
- 핵심 사용 흐름

마무리 (1문장):
- 출시 시기, 가격, 접근 방법
```

#### FAQ 구조

**고객 FAQ (External FAQ)** — 5개 이상

| 질문 유형 | 예시 |
|---|---|
| 이게 뭐예요? | "이 서비스는 정확히 무엇인가요?" |
| 왜 필요해요? | "기존 방법 대신 이걸 써야 하는 이유는?" |
| 어떻게 써요? | "시작하려면 어떻게 해야 하나요?" |
| 얼마예요? | "무료인가요? 유료라면 가격은?" |
| 불안해요 | "내 데이터는 안전한가요?" |

**내부 FAQ (Internal FAQ)** — 3개 이상

| 질문 유형 | 예시 |
|---|---|
| 시장 규모 | "이 시장은 얼마나 큰가요?" |
| 경쟁 | "경쟁 서비스와 차별점은?" |
| 수익 모델 | "어떻게 돈을 벌 건가요?" |
| 실행 가능성 | "혼자서/소규모로 만들 수 있나요?" |

### 4. AI를 활용한 PR&FAQ 작성 (10분)

#### 프롬프트 엔지니어링 기초

AI에게 좋은 결과물을 받기 위한 3가지 원칙:

1. **역할 부여**: "너는 스타트업 사업기획 전문가야"
2. **맥락 제공**: 문제, 타겟, 상황을 구체적으로 설명
3. **출력 형식 지정**: "아래 형식에 맞춰 작성해줘"

#### 실전 프롬프트 템플릿

```
너는 스타트업 사업기획 전문가야.

나는 [타겟 사용자]를 위한 [서비스 유형]을 기획하고 있어.

[문제 상황 설명]
- 구체적인 불편함 1
- 구체적인 불편함 2
- 현재 대안과 그 한계

[내가 생각하는 솔루션]
- 핵심 기능 1
- 핵심 기능 2

위 내용을 바탕으로 아마존 스타일 PR&FAQ를 작성해줘.

PR(보도자료)에는 다음을 포함해줘:
- 서비스명 제안
- 제목, 부제목
- 도입부, 문제, 솔루션, 고객 인용구, 사용 방법, 마무리

FAQ에는 다음을 포함해줘:
- 고객 FAQ 5개
- 내부 FAQ 3개
```

#### AI 결과물 검토 포인트

AI가 작성한 초안을 그대로 쓰면 안 됩니다. 다음을 반드시 확인하세요:

- [ ] **내 경험이 반영되었는가?** — AI는 일반적인 내용을 쓰는 경향이 있음
- [ ] **타겟 사용자가 구체적인가?** — "모든 사람"이 아닌 특정 그룹
- [ ] **문제가 진짜 문제인가?** — 내가 실제로 겪은 불편함인지 확인
- [ ] **솔루션이 현실적인가?** — 바이브 코딩으로 만들 수 있는 범위인지
- [ ] **숫자가 합리적인가?** — AI가 만든 통계나 수치는 검증 필요

### 5. PR&FAQ 예시 (10분)

#### 예시: "밥메이트" — 대학생 혼밥 매칭 서비스

**PR (보도자료)**

> **밥메이트, 캠퍼스 혼밥족을 위한 실시간 밥친구 매칭 서비스 출시**
>
> *같은 시간, 같은 식당, 새로운 친구 — 3초 만에 밥 약속 완성*
>
> 2026년 4월, 밥메이트가 이화여대 캠퍼스에서 베타 서비스를 시작한다. 밥메이트는 혼자 밥 먹기 싫지만 매번 약속 잡기는 부담스러운 대학생들을 위한 실시간 밥친구 매칭 서비스다.
>
> 대학내일 설문에 따르면 대학생의 67%가 일주일에 3회 이상 혼밥을 하며, 이 중 절반은 "같이 먹고 싶지만 약속 잡기 귀찮아서"라고 답했다. 기존 에브리타임 게시판에 글을 올리는 방식은 응답까지 시간이 걸리고, 어색한 DM을 보내야 하는 부담이 있다.
>
> 밥메이트는 "지금 밥 먹으러 갈래요" 버튼 하나로 작동한다. 현재 위치와 희망 식당을 선택하면, 같은 시간대에 밥 먹을 사람이 30초 내로 매칭된다. 매칭 후에는 간단한 프로필(학과, 학년, 관심사)만 공유되어 부담 없이 만남을 시작할 수 있다.
>
> 베타 테스터 김이화(경영학과 3학년) 씨는 "점심마다 에타에 글 올리던 게 이제 버튼 한 번이면 된다. 지난주에 매칭된 선배랑 스터디도 하게 됐다"고 말했다.
>
> 밥메이트 앱은 4월 중 이화여대에서 무료로 사용 가능하며, 웹 버전으로 먼저 출시된다.

**FAQ (발췌)**

- Q: 매칭된 사람이 안 나오면 어떻게 하나요?
  - A: 매칭 확정 후 10분 내 미도착 시 자동 취소되며, 노쇼 3회 시 이용이 제한됩니다.
- Q: 기존 에브리타임과 뭐가 다른가요?
  - A: 에타는 게시판 기반이라 글쓰기→대기→DM→약속 과정이 필요합니다. 밥메이트는 버튼 하나로 30초 내 매칭이 완료됩니다.
- Q: 어떻게 수익을 내나요?
  - A: 기본 매칭은 무료이며, 캠퍼스 주변 식당 제휴 광고와 프리미엄 매칭(관심사 기반)으로 수익을 창출합니다.

---

### 6. 전체 문서 흐름 이해하기 (10분)

#### 아이디어에서 개발까지, 문서의 여정

```
나의 문제  →  PR&FAQ  →  RFP  →  PRD  →  SPEC  →  바이브 코딩
(2주차)      (오늘)     (오늘)   (오늘)    (4주차)    (4주차~)
 "뭐가        "왜         "뭘      "어떤      "기술적     "만들자!"
  불편?"     만들어?"    만들어?"  기능?"    으로 어떻게?"
```

#### 문서별 역할 비교

| 문서 | 한 줄 정의 | 독자 | 핵심 질문 | 분량 |
|---|---|---|---|---|
| **PR&FAQ** | 가상 보도자료 + 예상 Q&A | 누구나 | "이거 왜 필요해?" | 2페이지 |
| **RFP** | 제안요청서 (무엇을 만들지) | 기획자/개발자 | "뭘 만들어야 해?" | 1~2페이지 |
| **PRD** | 제품요구사항 (어떤 기능이 필요한지) | 개발자/디자이너 | "어떤 기능이 있어?" | 3~5페이지 |
| **SPEC** | 기술 명세서 (어떻게 구현할지) | 개발자 | "기술적으로 어떻게?" | 5~10페이지 |

#### 왜 이 순서가 중요한가?

- **PR&FAQ 없이 RFP** → "왜 만드는지" 모른 채 요구사항 나열
- **RFP 없이 PRD** → 범위가 끝없이 늘어남 (스코프 크립)
- **PRD 없이 SPEC** → 기능 정의 없이 기술 설계 (삽질)
- **SPEC 없이 코딩** → AI에게 뭘 시켜야 할지 모름

> 바이브 코딩 시대에도 문서가 중요한 이유: **AI에게 좋은 프롬프트를 주려면, 명확한 문서가 필요합니다.**

### 7. RFP (제안요청서) 작성법 (10분)

#### RFP란?

- **Request for Proposal** (제안요청서)
- 원래는 "이런 시스템 만들어주세요"라고 외부에 요청하는 문서
- 스타트업에서는 **"우리가 무엇을 만들 것인지"를 정의하는 문서**로 활용

#### RFP 작성 템플릿

```markdown
# [서비스명] RFP

## 1. 프로젝트 개요
- 서비스 한 줄 설명
- 타겟 사용자
- 해결하려는 핵심 문제

## 2. 프로젝트 배경
- 왜 이 서비스가 필요한지 (PR&FAQ의 문제 부분 요약)
- 기존 대안의 한계

## 3. 목표
- 정량 목표: 출시 후 4주 내 사용자 100명 확보
- 정성 목표: "~한 경험을 제공한다"

## 4. 범위 (Scope)
### 포함 (In Scope)
- 반드시 포함할 기능/화면 (MVP 기준)

### 제외 (Out of Scope)
- 이번 버전에서는 만들지 않을 것

## 5. 일정
- 기획 완료: ~월 ~일
- 개발 시작: ~월 ~일
- MVP 배포: ~월 ~일

## 6. 기술 제약사항
- 웹/앱 여부
- 사용 도구 (Cursor, Claude 등)
- 배포 플랫폼 (Vercel, Netlify 등)
```

#### RFP의 핵심: "범위(Scope)"

가장 중요한 섹션은 **범위**입니다.

| 흔한 실수 | 올바른 접근 |
|---|---|
| "회원가입, 로그인, 프로필, 매칭, 채팅, 리뷰, 관리자 페이지, 알림..." | "매칭 + 결과 확인, 이 2가지만 먼저 만든다" |
| 모든 기능을 한 번에 | 핵심 1~2개만 MVP로 |
| Out of Scope 없음 | "채팅은 v2에서 추가" 명시 |

> **MVP 원칙**: 사용자가 핵심 가치를 경험할 수 있는 최소한의 기능만 만든다.

### 8. PRD (제품요구사항) 작성법 (15분)

#### PRD란?

- **Product Requirements Document** (제품요구사항서)
- RFP에서 정한 범위를 **구체적인 기능 목록**으로 풀어쓴 문서
- "무엇을 만들지"(RFP) → **"어떤 기능이 필요한지"**(PRD)

#### PRD의 3가지 핵심 구성요소

**1. 사용자 스토리 (User Story)**

"누가, 무엇을, 왜 하는가"를 한 문장으로 정리합니다.

> **형식**: "[사용자]로서, [행동]을 하고 싶다. 왜냐하면 [목적]이기 때문이다."

| ID | 사용자 | 행동 | 목적 |
|----|--------|------|------|
| US-01 | 혼밥하고 싶지 않은 학생 | 현재 위치에서 밥 메이트 찾기 | 같이 밥 먹을 사람을 빠르게 만나기 |
| US-02 | 매칭된 학생 | 상대방 프로필 확인하기 | 만남 전 기본 정보 파악 |
| US-03 | 재방문 학생 | 지난 매칭 기록 보기 | 다시 만나고 싶은 사람 찾기 |

> 팁: "모든 사용자"가 아니라 **구체적인 상황의 사용자**로 적는 것이 중요합니다.

**2. 화면 목록 (Screen List)**

서비스에 필요한 화면을 나열하고, 우선순위를 매깁니다.

| 우선순위 | 의미 | 기준 |
|----------|------|------|
| **Must** | MVP에 반드시 포함 | 이거 없으면 서비스가 안 됨 |
| **Should** | MVP에 포함하면 좋음 | 있으면 훨씬 나은 경험 |
| **Nice-to-have** | v2에서 추가 | 없어도 핵심 가치 전달 가능 |

예시 (밥메이트):

| 화면 | 설명 | 우선순위 |
|------|------|----------|
| 홈 | 매칭 시작 버튼, 현재 위치 표시 | Must |
| 매칭 결과 | 매칭된 상대 프로필 + 식당 정보 | Must |
| 프로필 설정 | 학과, 학년, 관심사 입력 | Should |
| 히스토리 | 지난 매칭 기록 | Nice-to-have |

> **MVP는 Must만 만든다.** Should, Nice-to-have는 출시 후 반응을 보고 추가합니다.

**3. 기능 상세 (Feature Detail)**

Must 화면의 각 기능을 입력 → 처리 → 출력 → 예외로 정리합니다.

```markdown
### F-01: 매칭 시작
- 입력: 희망 식당 영역, 식사 시간
- 처리: 같은 시간대 + 같은 영역 사용자 매칭
- 출력: 매칭 상대 정보, 식당 추천
- 예외: 매칭 상대 없을 시 "잠시 후 다시 시도" 안내

### F-02: 매칭 결과 확인
- 입력: 매칭 알림 탭
- 처리: 상대방 프로필(학과, 학년, 관심사) 조회
- 출력: 프로필 카드 + 식당 위치 지도
- 예외: 상대가 취소한 경우 "매칭이 취소되었습니다" 안내
```

#### PRD 작성 템플릿 (전체)

```markdown
# [서비스명] PRD

## 1. 개요
- 서비스 한 줄 설명
- 타겟 사용자
- 핵심 가치

## 2. 사용자 스토리
| ID | 사용자 | 행동 | 목적 |
|----|--------|------|------|
| US-01 | ... | ... | ... |

## 3. 화면 목록
| 화면 | 설명 | 우선순위 |
|------|------|----------|
| ... | ... | Must/Should/Nice-to-have |

## 4. 기능 상세 (Must 기능만)
### F-01: [기능명]
- 입력:
- 처리:
- 출력:
- 예외:

## 5. MVP 범위 요약
- 포함: Must 화면/기능 목록
- 제외: Should, Nice-to-have (향후 추가 예정)
```

#### AI로 PRD 작성하기

```
아래 PR&FAQ와 RFP를 바탕으로 PRD를 작성해줘.

[PR&FAQ 붙여넣기]
[RFP 붙여넣기]

조건:
- 사용자 스토리는 5개 이상
- 화면 목록에 우선순위(Must/Should/Nice-to-have) 반드시 포함
- Must 기능만 기능 상세(입력/처리/출력/예외) 작성
- MVP 범위는 Must 화면 2~3개로 제한
```

### 9. SPEC 미리보기 (5분)

#### SPEC (기술 명세서)

PRD의 기능을 **어떻게 구현할지** 기술적으로 정의한 문서입니다. 다음 주(4주차)에 AI를 활용해 PRD → SPEC 변환을 실습합니다.

```markdown
# [서비스명] 기술 SPEC

## 기술 스택
- Frontend: Next.js + Tailwind CSS
- Backend: Supabase (DB + Auth + API)
- 배포: Vercel
- AI 코딩: Cursor + Claude

## 데이터 모델
### users 테이블
| 필드 | 타입 | 설명 |
|------|------|------|
| id | uuid | 사용자 고유 ID |
| department | text | 학과 |
| grade | int | 학년 |
| interests | text[] | 관심사 배열 |

## API 설계
### POST /api/match
- 요청: { location, preferred_area, time_slot }
- 응답: { matched_user, restaurant_suggestion }
- 에러: 404 매칭 실패, 429 요청 제한 초과
```

> **다음 주 핵심**: PRD를 AI에게 주고 SPEC을 자동 생성한 뒤, 바로 바이브 코딩을 시작합니다.

핵심 포인트:
- PRD와 SPEC은 **AI에게 줄 프롬프트의 원재료**
- SPEC이 정확할수록 바이브 코딩의 결과물이 좋아짐
- "자연어로 코드를 작성한다" = "좋은 SPEC을 작성한다"

### 10. 실습: PR&FAQ → RFP → PRD 작성 (35분)

#### Step 1: 아이템 선정 (3분)
- 2주차 과제에서 정리한 "나의 문제 3가지" 중 하나를 선택
- 선택 기준:
  - 내가 가장 자주 겪는 문제
  - 나와 비슷한 사람들도 겪을 것 같은 문제
  - 앱이나 웹으로 해결 가능한 문제

#### Step 2: AI와 함께 PR&FAQ 초안 작성 (10분)
- Claude 또는 ChatGPT에 프롬프트 입력 (Part 1의 템플릿 활용)
- 결과물 검토 및 수정
- 내 경험과 구체적인 맥락 추가

#### Step 3: RFP 초안 작성 (7분)
- PR&FAQ를 기반으로 RFP 템플릿 채우기
- 특히 **범위(Scope)** 섹션에 집중
  - In Scope: 핵심 기능 1~2개만
  - Out of Scope: 나머지는 명확히 제외
- AI 프롬프트 예시:
  ```
  아래 PR&FAQ를 바탕으로 MVP용 RFP를 작성해줘.

  [PR&FAQ 붙여넣기]

  조건:
  - MVP는 핵심 기능 2개 이하로 제한
  - 혼자서 2주 내 바이브 코딩으로 만들 수 있는 범위
  - 웹 서비스 기준 (앱 아님)
  - Out of Scope도 반드시 포함
  ```

#### Step 4: PRD 초안 작성 (10분)
- RFP의 범위를 기반으로 PRD 템플릿 채우기
- 핵심 3가지에 집중:
  1. **사용자 스토리** 3~5개 작성
  2. **화면 목록**에 Must/Should/Nice-to-have 우선순위 매기기
  3. **Must 기능**만 상세(입력/처리/출력/예외) 작성
- AI 프롬프트 예시:
  ```
  아래 PR&FAQ와 RFP를 바탕으로 PRD를 작성해줘.

  [PR&FAQ 붙여넣기]
  [RFP 붙여넣기]

  조건:
  - 사용자 스토리 5개 이상
  - 화면 목록에 우선순위(Must/Should/Nice-to-have) 포함
  - Must 기능만 상세(입력/처리/출력/예외) 작성
  - MVP는 Must 화면 2~3개로 제한
  ```

#### Step 5: 동료 피드백 (5분)
- 옆 사람과 PRD 교환
- 피드백 포인트:
  - Must 기능만으로 핵심 가치를 경험할 수 있는가?
  - MVP 범위가 현실적인가? (2주 내 만들 수 있는가?)
  - 빠진 사용자 스토리나 예외 상황이 있는가?

## 11. 마무리 및 다음 주 안내 (5분)

### 오늘 배운 것
- PR&FAQ = 만들기 전에 가치를 검증하는 도구
- RFP = 무엇을 만들지 범위를 정하는 도구
- PRD = 어떤 기능이 필요한지 구체적으로 정의하는 도구
- 문서 흐름: PR&FAQ → RFP → PRD → SPEC → 바이브 코딩
- 오늘 PR&FAQ → RFP → PRD까지 완료, 다음 주에 SPEC 생성 후 바로 코딩 시작

### 다음 주 예고 (4주차)
- **PR&FAQ 간단 발표** (2~3분/인, 몇 명 선정)
- **강의/실습**: PRD → 기술 SPEC 자동 생성 (AI 활용)
- **실습**: 바이브 코딩 시작 — Cursor + Claude로 첫 번째 MVP 개발 착수

### 과제 (2가지 제출)
> **과제 1: PR&FAQ 제출**
> - 분량: PR 1페이지 + FAQ 1페이지 (총 2페이지 내외)
> - PR&FAQ를 주변 3명에게 보여주고 반응을 한 줄씩 기록해오기 (선택)
>
> **과제 2: PRD 제출**
> - 오늘 배운 PRD 템플릿에 맞춰 작성
> - 사용자 스토리, 화면 목록, 기능 상세 포함
> - AI 활용 작성 권장 (단, 본인이 검토/수정한 최종본 제출)
>

### PRD → SPEC → 코딩 전체 흐름 요약

```
[프롬프트 001] 자연어로 요구사항 나열
    ↓ AI가 PRD 생성
[프롬프트 002~003] 추가 요구사항/변경사항 반영
    ↓ PRD 수정
[프롬프트 004] "프루프리딩해줘" → AI가 7개 질문 → 의사결정 반영
    ↓ PRD 완성
[프롬프트 005] "PRD 토대로 SPEC 작성해줘" (한 줄!)
    ↓ SPEC 자동 생성
[프루프리딩] PRD↔SPEC 불일치 검토 → 수정
    ↓ SPEC 완성
[바이브 코딩] SPEC 기반으로 Cursor + Claude가 코드 생성
    ↓
[커밋 1e19397] 전체 기능 구현 완료!
```
