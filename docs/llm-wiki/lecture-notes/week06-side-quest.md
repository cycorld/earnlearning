## 이 체크리스트 사용법

내 페이스대로 한 단계씩 — **7단계 자율 학습 체크리스트**.
끝까지 안 가도 OK. 한 칸이라도 체크하면 그게 오늘의 진전입니다.

| 원칙 | 의미 |
|---|---|
| **01. AI 에게 시키는 작업** | 대부분 항목이 프롬프트 1~2개로 끝납니다. 직접 코딩할 일은 많지 않아요. |
| **02. 순서대로 가세요** | Lv 0 → Lv 6. 어려운 단계를 먼저 시도하면 앞 단계가 무너질 수 있습니다. |
| **03. 30분 룰** | 한 레벨에서 30분 이상 막히면 멈추세요. 다음 주 수업에서 함께 풀어요. |

## Roadmap — 7단계 챌린지 한눈에

| Lv | 단계 | 트랙 |
|---|---|---|
| **Lv 0** | 화면 마무리 | A 트랙 |
| **Lv 1** | 내 프로젝트 지도 | B 트랙 핵심 |
| **Lv 2** | 보안 & 권한 점검 | B 트랙 핵심 |
| **Lv 3** | GitHub & Vercel 배포 | C 트랙 |
| **Lv 4** | RBAC 리팩토링 | C 트랙 |
| **Lv 5** | 4역할 RBAC + 시드 | C 트랙 |
| **Lv 6** | 리팩토링 라운드 | C 트랙 |

> **트랙 매핑**: A → Lv 0 / B → Lv 1~2 / C → Lv 3~

---

## Lv 0 — 화면 마무리

> **A 트랙용** — 지난 주에 화면이 안 나온 사람

- [ ] 홈 화면 1개 완성
- [ ] 핵심 화면 1개 더 완성
- [ ] 두 화면 간 이동(네비게이션) 연결
- [ ] 로컬에서 `npm run dev` 로 확인

> 화면이 나오면 다음 단계로 — 다음 단계가 진짜 시작입니다.

---

## Lv 1 — 내 프로젝트 지도 그리기

> **B 트랙 핵심** — 이미 만든 것에 이름표 붙이기

- [ ] 프롬프트 카탈로그 ①번 (구조 파악) 실행 → 결과 저장
- [ ] AI 가 만들어준 CRUD 매핑 표를 스크린샷으로 캡처
- [ ] 내 프로젝트의 테이블 목록을 **3개 이상** 적어보기
- [ ] 각 화면이 어떤 CRUD 에 해당하는지 직접 손으로 체크
- [ ] "이건 내가 몰랐네" 싶은 지점 **1개 이상** 기록

> 이 단계가 진짜 핵심 — **"내 코드를 내가 모른다" 상태에서 벗어나는 순간.**

---

## Lv 2 — 보안 & 권한 점검

> AI 에게 내 코드를 털어보게 하기

- [ ] 프롬프트 카탈로그 ④번 (보안 점검) 실행
- [ ] **High** 위험도 이슈가 나오면 바로 AI 에게 고치라고 요청
- [ ] `.env.local` 이 `.gitignore` 에 있는지 확인 (없으면 추가)
- [ ] `service_role` 키가 클라이언트 코드에 없는지 확인
- [ ] Supabase RLS 정책 설정 여부 확인

> **IR Day 전에 보안 이슈는 반드시 한 번 털고 가세요.** 투자자 앞에서 키가 노출되면 안 됩니다.

---

## Lv 3 — GitHub 업로드 & Vercel 배포

> 내 코드를 인터넷 주소로 만들기

- [ ] 프롬프트 카탈로그 ⑥번 (배포 준비) 실행 → 빌드 에러 수정
- [ ] `git init` → `git add .` → `git commit -m "initial commit"`
- [ ] GitHub 에서 **Private** 저장소 생성 → push
  - [ ] `.env.local` 이 올라가 있지 않은지 확인 (올라갔다면 키 즉시 재발급!)
- [ ] [vercel.com](https://vercel.com) → GitHub 로그인 → Project Import
- [ ] **Environment Variables** — Production / Preview / Development 전부 체크
- [ ] **Deploy** 클릭 → 2~4분 대기
- [ ] 배포 URL 접속 → F12 콘솔 빨간 에러 없는지 확인

---

## Lv 4 — RBAC 리팩토링

> **C 트랙용** — 기존 권한 구조를 제대로 정리

- [ ] 현재 권한 체크가 어떻게 돼 있는지 ①번 프롬프트로 파악
- [ ] `if user.is_admin` 같은 **하드코딩된 체크**가 있는지 확인
- [ ] 프롬프트 카탈로그 ③번 (RBAC 리팩토링) 실행
- [ ] 역할 2개로 시작 (`admin` / `user`)
- [ ] 서버 사이드 권한 체크 미들웨어가 추가됐는지 확인
- [ ] UI 에서도 역할에 따라 버튼/메뉴가 숨겨지는지 확인
- [ ] **`user` 계정으로 관리자 API 직접 호출 시 403 반환되는지 검증**

> 마지막 검증 항목이 **진짜 RBAC 작동 여부**를 가르는 기준입니다.

---

## Lv 5 — 4역할 RBAC + 시드 데이터

> `admin / editor / author / viewer` 풀 RBAC

- [ ] 역할 4개로 확장: `admin` / `editor` / `author` / `viewer`
- [ ] AI 에게 "각 역할별 테스트 계정과 시드 데이터 만들어줘" 요청
- [ ] 각 계정으로 로그인해서 권한이 다르게 보이는지 **교차 확인**
- [ ] `author` 계정으로 남의 글은 수정 불가, 본인 글만 수정 가능한지 확인

> 여기까지 오면 **일반적인 CMS 수준의 권한 시스템이 완성**됩니다.

---

## Lv 6 — 리팩토링 라운드

> AI 가 만든 코드를 "왜 이렇게" 이해하기

- [ ] 프롬프트 카탈로그 ⑤번 (리팩토링) 을 가장 어지러운 파일에 실행
- [ ] 변경 전/후 diff 를 훑어보면서 "왜 이렇게 바뀌었는지" 한 줄씩 이해
- [ ] 이해 안 되는 부분은 AI 에게 **"이 줄은 왜 바뀐 거야?"** 다시 질문

> 마지막은 **"이해"** 의 단계 — 코드를 읽을 수 있게 되면 다음 학기가 완전히 달라집니다.

---

## The 30-Min Rule — 언제 멈출지 판단하는 법

> **30분**: 한 레벨에서 30분 이상 막히면 그 레벨에서 멈추세요.
> 막힌 채로 시간을 더 쓰는 것보다, 멈추고 다음 주에 함께 푸는 게 낫습니다.

| 막힘 ≠ 실력 부족 | 여기까지가 충분 | 다음 주 함께 해결 |
|---|---|---|
| 도구가 막힐 수도 있고 프롬프트 표현이 안 맞을 수도 있음 | 한 칸이라도 체크했다면 그게 오늘의 진전 | 막힌 부분은 다음 주 수업 시간에 같이 풀어요 |
