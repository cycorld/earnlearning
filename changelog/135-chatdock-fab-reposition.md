# 135. 챗봇 버튼이 전송버튼을 가리는 문제 — 숨김 + 드래그 이동

**날짜**: 2026-06-16
**티켓**: #135
**타입**: 기능 / UX 개선

## 무엇을 했나요?

화면 우하단에 항상 떠 있는 **챗봇 플로팅 버튼(FAB)** 이 가끔 페이지 자체의 전송버튼(예: 1:1 메시지 입력창)을 가리던 문제를 두 갈래로 고쳤습니다.

- **A. 상황에 맞춰 숨김**: 자기 입력창이 있는 화면(메시지 대화)에선 FAB를 아예 안 띄우고, 어떤 입력칸이든 **타이핑 중에는** FAB가 잠깐 사라집니다.
- **C. 드래그로 이동**: FAB를 끌어서 화면 **6개 모서리**(상단 좌/우, 중앙 좌/우, 하단 좌/우) 중 원하는 곳에 붙일 수 있고, 위치는 기억됩니다.

## 왜 필요했나요?

FAB는 `position: fixed` 로 모든 페이지 위에 겹쳐 떠 있습니다. 그런데 1:1 메시지 화면처럼 **페이지에도 우하단 전송버튼**이 있으면, 둘이 같은 자리에서 겹쳐 전송버튼을 누르기 어려웠습니다.

근본 원인은 "전역 고정 버튼"과 "페이지 버튼"의 자리 충돌이라, **충돌이 생기는 상황 자체를 없애는 것(A)** 을 1순위로, **사용자가 직접 자리를 바꿀 수 있는 자유도(C)** 를 더했습니다.

## 어떻게 만들었나요?

### A — 컨텍스트 숨김
- 라우트가 `/messages/:id`(대화)면 FAB를 렌더하지 않음.
- `document` 의 `focusin`/`focusout` 을 들어, 챗봇 밖 `input`/`textarea`/`select`/`contenteditable` 에 포커스가 가면 FAB를 `opacity-0 + pointer-events-none` 로 잠시 숨김.

### C — 6모서리 드래그
- 포인터 이벤트(마우스·터치 공용)로 끌고, 손을 떼면 **가장 가까운 모서리로 스냅**.
- **탭과 드래그 구분**: 8px 미만 움직임은 "탭"으로 보고 챗봇 열기, 그 이상은 "드래그"로 보고 이동(이때 뒤따르는 click 은 무시).
- 위치는 `localStorage('chatdock:fab-anchor')` 에 저장해 다음에도 유지. 모바일 스크롤 방해를 막으려 `touch-action: none`.
- 순수 함수 `nearestAnchor` / `anchorStyle` / `loadAnchor` / `saveAnchor` 로 로직을 분리(`fab-position.ts`).

## 배운 점

- **충돌은 "피하게 해주기"보다 "안 생기게 하기"가 먼저**: 드래그(C)는 자유도를 주지만 충돌 책임을 사용자에게 넘깁니다. 그래서 "자기 입력창 있는 화면에선 숨김"(A)을 핵심 픽스로 두고 드래그는 보조로 얹었습니다.
- **탭 vs 드래그**는 모든 끌 수 있는 버튼의 공통 난제 — 움직임 임계값(8px)과 "드래그 후 click 1회 무시" 두 장치로 깔끔히 갈립니다.
- 세로 중앙 정렬에 `transform: translateY(-50%)` 대신 `marginTop` 을 쓴 이유: hover 확대(`scale`)·누름(`translate-y`) 같은 다른 transform 과 충돌하지 않게 하기 위함.
