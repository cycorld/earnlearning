# 강의가 여러 개여도 경제는 강의별로 — 멀티 강의실 (지갑 분리 → 도메인 격리 → 컨텍스트 UI)

## 무엇을 했나요?

새 학기 새 강의를 같은 서비스에서 운영하기 위해 세 단계로 **강의실 격리**를 완성했습니다.

**1단계 — 지갑 분리**
- 한 학생이 여러 강의실에 속하면 강의실마다 독립된 지갑이 생깁니다.
- 강의실 초대 코드로 조인하면 그 강의실 전용 지갑에만 초기 자본금이 지급됩니다.
- "현재 보고 있는 강의실"(활성 강의실) 개념이 생겼고, `POST /classrooms/:id/activate`로 전환할 수 있습니다. 지갑 조회·랭킹·관리자 대시보드가 활성 강의실 기준으로 동작합니다.

**2단계 — 금융 도메인 격리**
- 회사·외주·대출·지원금에 `classroom_id`를 달았습니다. 생성 시 활성 강의실이 찍히고, 목록(회사·거래소 상장·투자 라운드·외주·지원금·내 대출)은 활성 강의실 것만 보입니다.
- 다른 강의실의 회사에 주문·투자하거나 외주·지원금에 지원하려 하면 `ErrWrongClassroom`으로 차단됩니다.
- 시간이 지나 실행되는 돈의 흐름(거래소 체결 정산, 배당, 라운드 취소 환불, 외주 에스크로·대금, 지원금 지급·회수, 대출 상환·주간이자, 청산 분배, 게시글·과제 보상, 송금)이 전부 "그 엔티티가 속한 강의실의 지갑"으로 정확히 들어가고 나갑니다.

**3단계 — 컨텍스트 UI**
- 헤더에 강의실 스위처: 복수 소속이면 드롭다운으로 전환.
- 온보딩 게이트: 승인된 학생이 아직 어느 강의실에도 없으면 초대 코드 입력 화면이 먼저 나옵니다.
- 관리자 첫 화면에 강의실 컨텍스트 선택 카드. 관리자는 멤버가 아닌 강의실도 전환해 관리할 수 있습니다.

## 왜 필요한가요?

기존 구조는 유저당 지갑이 하나뿐이었습니다(`wallets.user_id UNIQUE`). 이 상태로 새 강의를 시작하면 두 가지 사고가 납니다.

1. 지난 강의에서 모은 돈·자산이 새 강의에 그대로 넘어옵니다 — 게임 밸런스 붕괴.
2. 새 강의실에 조인할 때마다 같은 지갑에 초기 자본금이 **또** 입금됩니다 — 중복 지급.

## 어떻게 만들었나요?

핵심 난관은 SQLite가 `ALTER TABLE`로 UNIQUE 제약을 제거할 수 없다는 점입니다. 그래서 새 테이블을 만들어 데이터를 복사하는 "리빌드" 마이그레이션을 썼습니다.

- **id를 보존**하며 복사해 거래내역(`transactions.wallet_id`)의 참조가 그대로 유효합니다.
- 기존 지갑은 학생의 강의실 멤버십을 따라 자동 배정됩니다.
- 구 테이블은 삭제하지 않고 `wallets_legacy_159`로 보존합니다 (프로덕션 데이터 보호 규칙).
- 재미있는 함정: 외래키 검사가 켜진 상태에서 테이블 이름을 바꾸면 SQLite가 다른 테이블의 참조 선언까지 따라 바꿔버립니다. 거래내역이 백업 테이블을 가리키게 되는 사고를 **테스트가 먼저 잡아냈고**, 마이그레이션 중에만 외래키 검사를 끄는 것으로 해결했습니다.

TDD로 진행했습니다: 1단계는 실패하는 테스트 5개(강의실별 잔액 분리, 중복 지급 방지, 비멤버 전환 차단, 강의실별 랭킹, 마이그레이션 데이터 보존), 2단계는 격리 테스트 7건(목록 필터, 타 강의실 액션 차단, 비동기 자금 흐름 라우팅)을 먼저 작성하고(Red) 구현으로 통과시켰습니다(Green). 전체 통합 테스트 369개 회귀 통과, 프론트엔드 vitest 210개 통과.

마지막으로 헤드리스 브라우저로 **E2E 검증**을 했습니다: 가상 학생 계정을 만들어 강의실 두 곳에 걸쳐 실제 UI로 가입→승인→코드 입장→회사 설립→강의실 전환을 돌려봤고, 여기서 누수 2건을 잡았습니다. ① 자산 화면의 주식 가치·회사 지분·부채가 강의실 필터 없이 집계되어 A 강의실 회사가 B 자산에 보이던 문제(현금만 스코프되어 있었음), ② 이미 강의실에 속한 학생이 두 번째 강의실에 들어갈 UI가 아예 없던 문제. 각각 회귀 테스트를 먼저 쓰고(Red) 고쳤습니다(Green) — 자산 집계 강의실 스코프, `/companies/mine` 스코프, 헤더 스위처에 "초대 코드로 입장" 추가.

2단계에서 제일 까다로웠던 건 "나중에 실행되는 돈"입니다. 주문을 넣을 때와 체결될 때의 활성 강의실이 다를 수 있으니, "지금 사용자가 보고 있는 강의실 지갑"이 아니라 **"그 회사(엔티티)가 속한 강의실의 지갑"**을 기준으로 입출금하도록 자금 흐름 하나하나를 라우팅했습니다. 1단계 시점에 한계로 남겨뒀던 "도메인 보상이 활성 지갑으로 들어가는 문제"가 이걸로 해소됐습니다.

## 사용한 프롬프트

> 회원가입 후 초대 코드에 맞춰서 새로운 강의에 학생들이 입장가능해야해 (기존 유저도 초대받을 수 있는데, 강의 마다 지갑 등은 구분되어야해 - 구현되어 있는지 검증해줘)

> 좋아. 설계안 확정화고 구현들어가자. TDD 방식으로 테스트 확실하게 하고 진행해야 해.

> 한번에 모두 작업하고 머지하자. phase2, phase3 진행 후에 모든 기능이 잘 동작하는지 가상계정들 만들고 모두 검증해줘. 여러 강의 실에 걸쳐서.

## 배운 점

- 스키마 제약을 바꾸는 마이그레이션은 "새 테이블 복사"가 정석이지만, 외래키·AUTOINCREMENT·이름 변경의 부수효과가 많아 **마이그레이션 자체를 테스트**해야 안전합니다.
- 인터페이스 시그니처를 유지한 채 내부 의미를 바꾸면(`FindByUserID` = 활성 강의실 지갑) 호출부 13곳을 건드리지 않고 격리를 도입할 수 있습니다. 대신 의미 변화를 문서화하지 않으면 나중에 함정이 됩니다.
