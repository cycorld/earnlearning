## 무엇을 했나요?

EarnLearning LMS 프로젝트의 기획과 설계를 진행했습니다. 구체적으로는:

- **PRD(Product Requirements Document)** 작성: 프로젝트가 왜 필요한지, 어떤 기능이 있어야 하는지 정리
- **기술 스택 결정**: Go + React + SQLite + Docker 조합 선택
- **인증/승인 시스템 설계**: 이메일 회원가입 + 관리자 승인 방식
- **다중 회사 설립 모델 설계**: 학생들이 가상 회사를 만들고 기업가치를 산정하는 로직
- **기술 스펙 문서(SPEC.md)** 작성: 개발자가 바로 구현할 수 있는 수준의 상세 명세
- **PWA + Web Push 알림 스펙** 추가: 모바일에서도 앱처럼 사용 가능하도록 설계
- **프루프리딩**: PRD와 SPEC 간 불일치 해결, 용어 통일, 로직 오류 수정

총 8개의 커밋으로 약 하루 동안 기획과 설계를 완료했습니다.

## 왜 필요했나요?

### 1. "코드부터 짜면 안 되나요?"

많은 초보 개발자들이 아이디어가 떠오르면 바로 코드를 작성하려 합니다. 하지만 실제 스타트업에서 이렇게 하면 다음과 같은 문제가 발생합니다:

- **방향 없는 개발**: 무엇을 만들어야 하는지 모르니 이리저리 헤맴
- **재작업**: "아, 이 기능도 필요했네?" 하며 이미 만든 것을 뜯어고침
- **소통 불가**: 팀원에게 "이거 만들어주세요" 하면 서로 다른 것을 상상함

PRD는 이 모든 문제를 예방합니다. **"우리가 무엇을 왜 만드는가"**를 명확히 합니다.

### 2. 왜 기술 스펙이 따로 필요한가?

PRD가 "무엇을(What)"에 대한 문서라면, 기술 스펙은 "어떻게(How)"에 대한 문서입니다.

예를 들어 PRD에 "사용자는 로그인할 수 있다"라고 적혀 있다면, 기술 스펙에는:
- JWT 토큰을 사용한다
- 토큰 만료 시간은 24시간이다
- Refresh Token은 7일이다
- 비밀번호는 bcrypt로 해시한다

이런 구체적인 구현 방법이 적혀 있습니다.

### 3. 왜 프루프리딩이 필요한가?

문서를 한 번에 완벽하게 쓸 수 있는 사람은 없습니다. 특히 PRD와 SPEC이 분리되어 있으면 **서로 모순**되는 내용이 생기기 쉽습니다. 예를 들어:

- PRD: "팀은 최대 5명"
- SPEC: "팀원 수 제한 없음"

이런 불일치를 발견하고 수정하는 과정이 프루프리딩입니다.

## 어떻게 만들었나요?

### 1단계: PRD 작성

PRD는 다음 구조로 작성했습니다:

```
1. 프로젝트 개요 (왜 만드는가?)
2. 타겟 사용자 (누구를 위한 것인가?)
3. 핵심 기능 목록 (무엇을 할 수 있는가?)
4. 유저 플로우 (사용자가 어떤 순서로 사용하는가?)
5. 성공 지표 (어떻게 성공을 측정할 것인가?)
```

이 프로젝트에서 특히 중요했던 결정:

- **게임화 요소**: 단순한 LMS가 아니라, 학생들이 가상 회사를 설립하고 운영하는 경험을 제공
- **기업가치 산정**: 학생들의 활동(과제 제출, 피드백, 마켓 참여)이 가상 기업가치에 반영

### 2단계: 기술 스택 선정

기술 스택을 고르는 것은 집을 지을 때 재료를 고르는 것과 같습니다.

| 영역 | 선택 | 이유 |
|------|------|------|
| 백엔드 | Go (Echo) | 빠르고 가벼움, 단일 바이너리 배포, 동시성 처리 우수 |
| 프론트엔드 | React + TypeScript | 생태계가 크고, 취업 시장에서 수요 높음 |
| 데이터베이스 | SQLite | 서버 한 대 운영에 최적, 설정이 간단, 백업 용이 |
| 배포 | Docker Compose | 환경 차이 없이 동일하게 실행, stage/prod 분리 용이 |
| CSS | Tailwind CSS | 클래스명만으로 빠르게 스타일링, 일관된 디자인 시스템 |
| UI 컴포넌트 | shadcn/ui | 복사해서 쓰는 방식으로 커스터마이징 자유로움 |

처음에는 다른 기술 스택을 검토했지만, **교육용 프로젝트의 특성**을 고려해 최종 결정을 내렸습니다:
- 학생들이 코드를 읽고 이해할 수 있어야 함
- 단일 서버에서 운영 가능해야 함
- 비용이 최소화되어야 함

### 3단계: 핵심 모델 설계

**다중 회사 설립 모델**은 이 프로젝트의 핵심입니다:

```
학생 → 회사 설립 (최대 N개)
     → 각 회사에 팀원 초대
     → 과제 수행 → 기업가치에 반영
     → 마켓에서 제품/서비스 거래
     → 기업가치 순위 경쟁
```

기업가치 산정 로직은 여러 요소를 종합적으로 반영합니다:
- 과제 완료율
- 팀원 활동도
- 마켓 거래 실적
- 피드백 품질

### 4단계: PWA 설계

PWA(Progressive Web App)는 웹사이트를 모바일 앱처럼 사용할 수 있게 해주는 기술입니다.

```
일반 웹사이트           PWA
────────────          ────────────
브라우저에서만 접근      홈 화면에 설치 가능
인터넷 필수            오프라인 캐시
알림 불가              Web Push 알림
주소창 보임            앱처럼 전체화면
```

교육 LMS에서 PWA가 중요한 이유:
- 학생들이 알림을 받아야 과제 마감일을 놓치지 않음
- 모바일에서 빠르게 접근 가능해야 참여율이 높아짐
- 별도 앱 개발 비용 절감

### 5단계: 프루프리딩

Claude에게 PRD와 SPEC을 동시에 읽게 하고, 다음을 점검했습니다:

1. **용어 통일**: 같은 개념을 다른 단어로 쓴 곳 수정
2. **로직 일관성**: PRD의 비즈니스 규칙과 SPEC의 기술 규칙이 일치하는지 확인
3. **누락 항목**: PRD에는 있지만 SPEC에 빠진 기능, 또는 그 반대
4. **모순 해결**: 서로 충돌하는 요구사항을 발견하고 의사결정

## 사용한 프롬프트

### PRD 작성 프롬프트
```
이화여자대학교 "스타트업을 위한 코딩입문" 강의용 게임화 LMS의 PRD를 작성해줘.
핵심 컨셉은 학생들이 가상 회사를 설립하고 운영하면서 코딩과 창업을 동시에 배우는 것이야.
```

### 기술 스택 변경 프롬프트
```
기술 스택을 재검토해줘. 단일 EC2 서버에서 운영할 건데,
학생 50명 규모에 맞는 가장 효율적인 조합이 뭘까?
```

### 프루프리딩 프롬프트
```
PRD.md와 SPEC.md를 동시에 읽고, 불일치하는 부분을 모두 찾아서 수정해줘.
용어도 통일하고, 로직 오류도 잡아줘.
```

### PWA 스펙 추가 프롬프트
```
PWA + Web Push 알림 스펙을 SPEC.md에 추가해줘.
VAPID 키 기반 인증을 사용하고, 오프라인 캐시 전략도 포함해줘.
```

프롬프트를 쓸 때 중요한 점은 **구체적인 제약 조건**을 함께 제시하는 것입니다. "LMS 만들어줘"보다 "학생 50명 규모, 단일 EC2, 게임화 요소 포함"이라고 하면 훨씬 더 정확한 결과를 얻을 수 있습니다.

## 배운 점

### 1. 기획이 개발 시간의 절반이다
PRD와 SPEC을 꼼꼼히 작성하면 실제 코딩할 때 "뭘 만들어야 하지?" 고민이 사라집니다. 이 프로젝트에서는 기획에 하루를 투자했고, 덕분에 전체 구현을 하루 만에 완료할 수 있었습니다.

### 2. 문서 간 정합성이 중요하다
PRD와 SPEC은 별도 문서이지만 하나의 제품을 설명합니다. 둘 사이에 모순이 있으면 개발자가 혼란에 빠집니다. 프루프리딩은 선택이 아니라 필수입니다.

### 3. 기술 스택 선택은 트레이드오프다
"최고의 기술"은 없습니다. 프로젝트의 규모, 팀의 역량, 예산, 일정에 따라 최적의 선택이 달라집니다. 이 프로젝트에서 SQLite를 선택한 것은 "50명 규모에서 가장 효율적"이기 때문이지, PostgreSQL보다 우수해서가 아닙니다.

### 4. PWA는 비용 대비 효과가 크다
네이티브 앱(iOS/Android)을 별도로 개발하면 비용이 2~3배 늘어납니다. PWA는 웹 기술만으로 앱과 유사한 경험을 제공하므로, 스타트업 초기에 특히 유용한 전략입니다.

### 5. AI를 활용한 프루프리딩
Claude 같은 AI 도구는 두 문서를 동시에 비교하며 불일치를 찾는 데 매우 효과적입니다. 사람이 하면 놓치기 쉬운 작은 모순도 체계적으로 발견할 수 있습니다.
