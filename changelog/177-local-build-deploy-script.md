# 자기 자신에게 SSH 접속하는 스크립트 — 배포 스크립트가 실행 위치를 착각할 때

## 무엇을 했나요?

배포 스크립트 `deploy-remote.sh`가 **빌드서버 cycorld 자신에서** 곧바로 돌도록 고쳤습니다. 예전엔 "내 노트북에서 실행 → 빌드서버로 `ssh cycorld` 접속 → 거기서 빌드" 흐름이었는데, 이제는 스크립트가 이미 빌드서버 위에 있으니 **접속 없이 그 자리에서 바로 빌드**합니다.

여기에 세 가지를 더 붙였습니다:

- **빌드 전 안전 검증**: 지금 브랜치가 `main`인지, 커밋 안 된 변경은 없는지, 원격 `origin/main`과 같은 커밋인지 확인. 하나라도 어긋나면 빌드를 **안 하고** 멈춥니다.
- **promote 태그 수정**: Prod로 올릴 버전을 "추측"하지 않고, Stage에서 **실제 돌고 있는 컨테이너에게 물어봐서** 그 버전을 올립니다.
- **회귀 테스트**: `deploy/tests/test-deploy-remote.sh` — 셸 스크립트인데도 자동 테스트를 붙였습니다.

> SSH(Secure Shell): 다른 컴퓨터에 원격으로 접속해 명령을 실행하는 도구. `ssh cycorld`는 "cycorld라는 별명의 서버에 접속" 이라는 뜻.

## 왜 필요했나요?

스크립트가 **어디서 실행되는지**를 착각하고 있었습니다.

처음 만들 때는 "개발자 노트북에서 실행"을 가정했습니다. 그래서 무거운 빌드를 하려고 `ssh cycorld`로 빌드서버에 접속하는 코드가 들어 있었죠. 그런데 운영 방식이 바뀌어서 **스크립트를 빌드서버 cycorld에서 직접** 돌리게 됐습니다. 문제는 cycorld 안에는 `cycorld`라는 SSH 별명이 없다는 것 — 즉 **자기 자신에게 접속하려다 실패**해서 배포가 시작조차 안 됐습니다.

또 하나, `promote`(Stage에서 확인된 버전을 Prod로 올리는 단계)에 숨은 버그가 있었습니다. 올릴 버전을 정할 때 **EC2 서버의 git 최신 커밋**을 읽어서 "아마 이 버전이 Stage에 떠 있겠지" 하고 **추측**했습니다. 하지만 git 커밋과 실제 배포된 이미지는 다를 수 있습니다 (누가 다른 걸 올렸거나, git이 앞서 있거나). 추측이 틀리면 **검증 안 된 버전이 프로덕션에 올라가는** 사고로 이어집니다.

> GHCR(GitHub Container Registry): 빌드한 도커 이미지를 보관하는 창고. 빌드서버가 여기 올리면 EC2가 여기서 내려받습니다.
>
> Blue-Green 배포: 똑같은 서버를 둘(blue/green) 두고, 새 버전을 노는 쪽에 먼저 띄운 뒤 트래픽만 휙 넘기는 방식. 실패하면 즉시 되돌릴 수 있어 무중단.

## 어떻게 만들었나요?

**1) 접속 대신 로컬 빌드.** `ssh cycorld` 한 줄을 걷어내고, 그 자리에서 `./deploy/build-and-push.sh`를 직접 부르게 했습니다. Stage/Prod 배포·롤백·상태 확인은 여전히 EC2로 가야 하므로 `ssh earnlearning` 경로는 그대로 뒀습니다.

**2) 순진한 수정은 위험하다.** "그럼 빌드 전에 자동으로 `git pull` 해서 최신으로 맞추면 되지 않나?" 싶지만, 이건 위험합니다.

- 개발자가 다른 브랜치에서 작업 중이었다면 그 브랜치를 **건드려 망가뜨릴** 수 있고,
- pull이 조용히 실패하면 **엉뚱하거나 오래된 코드**를 빌드해 배포할 수 있습니다.

그래서 자동으로 고치는 대신 **어긋나면 딱 멈추게(fail-fast)** 했습니다. main이 아니거나, 커밋 안 된 변경이 있거나, `origin/main`과 다르면 빌드 전에 실패하고 **"직접 `git pull --ff-only` 하세요"** 라고 안내만 합니다. *암묵적으로 알아서 해주기(implicit)* 보다 *명시적으로 사용자가 확인(explicit)* 하게 하는 쪽이 배포처럼 되돌리기 어려운 작업에선 훨씬 안전합니다.

**3) 추측하지 말고 물어봐라.** `promote`가 git HEAD를 읽던 부분을, Stage에서 실제 돌고 있는 컨테이너에게 직접 묻게 바꿨습니다:

```bash
ssh earnlearning "sudo docker inspect --format '{{.Config.Image}}' earnlearning-stage-backend-1"
```

여기서 이미지 태그만 뽑아 그대로 Prod에 올립니다. 태그가 비어 있거나 `latest`(어느 버전인지 알 수 없는 떠도는 태그)면 거부합니다. 핵심 교훈은 **"배포된 것을 물어봐라, 추측하지 마라."**

**4) 셸 스크립트도 테스트한다.** 셸 스크립트는 테스트하기 까다롭습니다. 진짜로 빌드하고 진짜로 SSH를 쏘면 안 되니까요. 그래서 **PATH 트릭**을 썼습니다.

> PATH: 셸이 명령어(예: `git`, `ssh`)를 찾을 때 뒤지는 폴더 목록. 앞쪽 폴더부터 먼저 찾습니다.

테스트에서는 `git`·`ssh`·`build-and-push.sh`라는 이름의 **가짜 실행파일**을 임시 폴더에 만들고, 그 폴더를 PATH 맨 앞에 끼웁니다. 그러면 스크립트가 진짜 대신 **가짜를 부르고**, 가짜는 "누가 나를 어떤 인자로 불렀는지"를 로그 파일에 적습니다. 실행이 끝나면 그 로그를 검사해서 증명합니다:

- `ssh cycorld`를 **한 번도 안 부른다** (자기 접속 버그 재발 방지)
- Stage에는 빌드된 태그가 넘어간다
- `promote`는 Stage 컨테이너 태그를 읽는다 (EC2 git HEAD가 아니라)
- dirty·비-main·비동기화 상태면 빌드 전에 실패한다

실행: `bash deploy/tests/test-deploy-remote.sh`

## 배운 점

- **스크립트는 "어디서 실행되는가"에 대한 숨은 가정을 갖는다.** 노트북용으로 짠 코드를 서버에서 돌리면, `ssh cycorld` 같은 한 줄이 조용히 무너집니다. 실행 환경이 바뀌면 그 가정부터 다시 봐야 합니다.
- **상태를 추측하지 말고, 실제로 돌고 있는 것에게 물어봐라.** git 커밋은 "무엇을 배포했을 것 같다"는 추측일 뿐이고, 컨테이너에게 묻는 건 사실입니다. 특히 프로덕션에선 추측이 곧 사고입니다.
- **자동으로 고쳐주는 것(implicit)보다 멈추고 알려주는 것(explicit)이 안전할 때가 있다.** 되돌리기 어려운 작업일수록 fail-fast가 낫습니다.
- **셸 스크립트도 회귀 테스트를 받을 자격이 있다.** PATH로 명령어를 가짜로 바꿔치기하면, 진짜 배포 없이도 스크립트의 행동을 검증할 수 있습니다.
