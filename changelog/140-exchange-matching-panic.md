# 140. 거래소 체결 시 서버가 죽던 버그 (그리고 "돈은 갔는데 주식은 안 오던" 데이터 손상)

**날짜**: 2026-06-30
**티켓**: #140
**타입**: 버그수정 (데이터 무결성)

## 무엇을 했나요?

거래소에서 **매수·매도 주문이 서로 맞아 체결되는 순간** 서버가 500 에러를 내며 죽던 버그를 고쳤습니다. 게다가 이 버그는 단순 에러가 아니라 **돈은 이동했는데 주식 소유권은 안 넘어가는** 데이터 손상을 남기고 있었습니다.

## 왜 필요했나요?

#139 거래소 개편을 스테이지에서 검증하다 발견했습니다. 두 사람이 같은 가격에 매수/매도를 걸어 체결시키면 매수자에게 `500` 이 떴습니다. 로그를 보니 서버가 패닉(panic, 프로그램이 더 진행 못 하고 멈추는 치명적 오류)으로 죽고 있었습니다.

핵심은 **"처음 그 회사 주식을 사는 사람"** 이었습니다.

체결이 일어나면 서버는 ① 체결 기록을 남기고 ② 사는 사람 지갑에서 돈을 빼 파는 사람에게 주고 ③ 주식 소유권을 옮깁니다. 그런데 ③에서 패닉이 나면, 트랜잭션(transaction, 여러 작업을 "전부 성공 아니면 전부 취소"로 묶는 장치)이 없던 탓에 **①②는 이미 저장된 채로** 멈췄습니다. 결과: 체결 기록과 돈 이동은 남고, **주식은 안 넘어간** 반쪽 상태.

## 어떻게 만들었나요?

### 근본 원인 — `(nil, nil)` 이라는 함정
주주를 찾는 `FindShareholder()` 는 **못 찾으면 에러가 아니라 `(nil, nil)`**(값도 없고 에러도 없음)을 돌려줍니다. "없음"을 표현하는 방식이죠.

그런데 체결 처리 코드(`runMatching`)의 매수자 분기는 이렇게 돼 있었습니다:

```go
buyerSH, err := FindShareholder(...)
if err != nil {           // 처음 사는 사람은 err==nil 이라 여기로 안 옴
    // 신규 주주 생성
} else {
    newShares := buyerSH.Shares + ...   // buyerSH 가 nil → 💥 패닉
}
```

처음 사는 사람은 주주 레코드가 없어 `(nil, nil)` → `err != nil` 이 거짓 → `else` 로 들어가 **nil 을 역참조**하며 죽었습니다. 매도자 분기에도 같은 지뢰가 있었습니다.

### 수정
- 매수자: `if err != nil` → `if err != nil || buyerSH == nil` (없으면 생성)
- 매도자: 조건에 `sellerSH != nil` 가드 추가

`FindShareholder` 자체를 바꾸지 않은 이유: `(nil, nil)` "없음" 신호를 **이미 올바르게 처리하는 다른 호출부**(예: 매도 가능 주식 검증)가 있어서, 함수를 바꾸면 그쪽이 깨집니다. 그래서 깨진 호출부만 고치는 **외과적 수정**을 택했습니다.

## 배운 점

- **"못 찾으면 `(nil, nil)`" 패턴은 호출부가 `err` 만 보면 터진다.** Go 에서 not-found 를 에러 대신 nil 로 표현하면, 받는 쪽은 **값의 nil 여부도 같이** 봐야 합니다. `err == nil` 이 "값이 있다"를 보장하지 않습니다.
- **돈을 옮기는 작업은 트랜잭션으로 묶어야 한다.** 이번엔 패닉만 막아 체결이 끝까지 진행되게 했지만, 중간 어느 단계가 실패해도 "전부 취소"되도록 `runMatching` 전체를 DB 트랜잭션으로 감싸는 후속 작업을 남겼습니다. (별도 티켓)
- **배포 검증은 실데이터로 끝까지 해보자.** 단위 테스트는 통과했지만, 스테이지에서 실제로 두 사람이 체결을 일으켜 보고서야 드러난 버그입니다.
