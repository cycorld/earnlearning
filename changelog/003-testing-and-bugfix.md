## 무엇을 했나요?

전체 기능 구현 후 발견된 버그들을 수정하고 테스트를 작성했습니다:

- **E2E(End-to-End) 테스트**: 사용자 시나리오 기반 50개 이상의 테스트 케이스 작성 및 버그 수정
- **마이너 버그 수정**: 강의실 초대코드 표시 오류, KPI 404 에러, Router 경고 등
- **댓글 버그 수정**: 댓글 작성/조회 관련 로직 오류 해결
- **스모크 테스트 추가**: 서버가 기본적으로 동작하는지 빠르게 확인하는 테스트
- **회귀 테스트 추가**: 수정한 버그가 다시 발생하지 않도록 방지하는 테스트

## 왜 필요했나요?

### "코드가 동작한다"와 "코드가 올바르게 동작한다"는 다르다

전체 기능을 구현한 직후의 상태를 솔직하게 말하면:

```
- 서버가 시작되긴 함 ✅
- 기본적인 API는 응답함 ✅
- 하지만 실제 시나리오에서 수많은 버그 발생 ❌
  - 초대코드가 화면에 표시 안 됨
  - KPI 페이지가 404
  - 댓글이 저장되지만 조회가 안 됨
  - 라우터 경고가 콘솔에 표시됨
```

이것은 AI가 생성한 코드의 문제가 아닙니다. **모든 소프트웨어에는 버그가 있습니다.** 중요한 것은 버그를 빠르게 발견하고 수정하는 체계를 갖추는 것입니다.

### 테스트가 없으면 생기는 문제

```
시나리오: "댓글 버그를 수정했다!"
         ↓
    하지만 댓글 수정이 회사 페이지를 망가뜨렸다
         ↓
    회사 페이지를 고쳤더니 마켓이 안 된다
         ↓
    "도대체 뭘 건드린 거지...?" 😱
```

이런 연쇄 반응을 **회귀(Regression)**라고 합니다. 테스트는 이런 회귀를 자동으로 감지해줍니다.

## 어떻게 만들었나요?

### 테스트의 종류 이해하기

소프트웨어 테스트는 피라미드 형태로 구성됩니다:

```
        /\
       /  \        E2E 테스트 (전체 흐름)
      /    \       - 느리지만 사용자 경험을 직접 검증
     /------\
    /        \     통합 테스트 (모듈 간 연동)
   /          \    - API 요청 → DB 저장 → 응답 확인
  /────────────\
 /              \  단위 테스트 (개별 함수)
/________________\ - 빠르고 많이 작성
```

이 프로젝트에서는 **통합 테스트**와 **스모크 테스트**에 집중했습니다.

### 스모크 테스트란?

"스모크 테스트"라는 이름은 전자 제품에서 유래했습니다. 새 회로 기판에 전원을 넣었을 때 **연기(smoke)가 나지 않으면** 기본적으로 동작한다는 뜻입니다.

소프트웨어에서 스모크 테스트는 "가장 기본적인 것이 동작하는가?"를 확인합니다:

```go
// 스모크 테스트 예시
func TestSmoke(t *testing.T) {
    // 1. 서버가 시작되는가?
    server := setupTestServer()
    defer server.Close()

    // 2. 헬스체크 API가 응답하는가?
    resp, err := http.Get(server.URL + "/api/health")
    assert.NoError(t, err)
    assert.Equal(t, 200, resp.StatusCode)

    // 3. 회원가입이 되는가?
    resp, err = http.Post(server.URL + "/api/auth/register", ...)
    assert.NoError(t, err)
    assert.Equal(t, 201, resp.StatusCode)

    // 4. 로그인이 되는가?
    resp, err = http.Post(server.URL + "/api/auth/login", ...)
    assert.NoError(t, err)
    assert.Equal(t, 200, resp.StatusCode)
}
```

스모크 테스트를 실행하는 명령어:
```bash
cd backend && go test ./tests/integration/ -run TestSmoke -timeout 60s
```

이 테스트가 실패하면 **다른 어떤 것도 동작하지 않을 것**이므로, 항상 먼저 실행합니다.

### E2E 테스트 시나리오

E2E 테스트는 실제 사용자의 행동을 시뮬레이션합니다:

```
시나리오 1: 신규 사용자 가입부터 회사 설립까지
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. 회원가입 페이지에서 이메일/비밀번호 입력
2. 관리자가 승인
3. 로그인
4. 대시보드 확인
5. "회사 설립" 클릭
6. 회사명, 설명 입력
7. 회사가 생성되었는지 확인

시나리오 2: 과제 제출 및 평가
━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. 로그인
2. 과제 목록 확인
3. 과제 상세 페이지 진입
4. 과제 내용 작성 및 제출
5. 제출 상태가 "제출됨"으로 변경 확인
6. 관리자 로그인
7. 제출된 과제 확인 및 평가

시나리오 3: 마켓 거래
━━━━━━━━━━━━━━━━━━━━
1. 상품 등록
2. 다른 사용자로 로그인
3. 상품 검색/조회
4. 구매
5. 기업가치 변동 확인
```

총 50개 이상의 시나리오를 작성하며, 각 시나리오에서 발견된 버그를 즉시 수정했습니다.

### 발견된 버그와 수정 방법

#### 버그 1: 강의실 초대코드가 표시되지 않음

```
원인: API 응답에 초대코드 필드가 포함되어 있지만,
     프론트엔드에서 해당 필드를 렌더링하지 않음

수정: 강의실 상세 페이지 컴포넌트에 초대코드 표시 추가
```

이런 버그는 백엔드와 프론트엔드를 각각 만들 때 자주 발생합니다. API는 데이터를 보내고 있지만, UI에서 그 데이터를 보여주지 않는 것이죠.

#### 버그 2: KPI 페이지 404 에러

```
원인: 라우터에 KPI 페이지 경로가 등록되지 않음
     (백엔드 API는 있지만 프론트엔드 라우팅이 누락)

수정: React Router에 /admin/kpi 경로 추가
```

#### 버그 3: 댓글이 저장되지만 조회 안 됨

```
원인: 댓글 조회 API의 SQL 쿼리에서 JOIN 조건 오류
     (작성자 정보를 가져오는 부분이 잘못됨)

수정 전: SELECT * FROM comments WHERE post_id = ?
수정 후: SELECT c.*, u.name as author_name
         FROM comments c
         JOIN users u ON c.user_id = u.id
         WHERE c.post_id = ?
```

### 회귀 테스트 작성

버그를 수정한 후, **같은 버그가 다시 발생하지 않도록** 테스트를 작성합니다:

```go
// 회귀 테스트: 댓글 조회 시 작성자 이름이 포함되어야 함
func TestCommentIncludesAuthorName(t *testing.T) {
    // 1. 테스트 사용자 생성
    user := createTestUser(t, "testuser@test.com")

    // 2. 게시글 작성
    post := createTestPost(t, user.ID, "테스트 글")

    // 3. 댓글 작성
    createTestComment(t, user.ID, post.ID, "테스트 댓글")

    // 4. 댓글 조회
    comments := getComments(t, post.ID)

    // 5. 작성자 이름이 포함되어 있는지 확인
    assert.Equal(t, "testuser", comments[0].AuthorName)
    // 이 테스트가 실패하면 → 댓글 조회 버그가 재발한 것!
}
```

### 테스트 자동화

모든 테스트는 하나의 명령어로 실행됩니다:

```bash
# 스모크 테스트 (가장 먼저)
cd backend && go test ./tests/integration/ -run TestSmoke -timeout 60s

# 전체 테스트
cd backend && go test ./... -timeout 120s
```

이 명령어를 커밋 전에 항상 실행하는 것이 규칙입니다.

## 사용한 프롬프트

### E2E 테스트 프롬프트
```
SPEC.md의 모든 사용자 시나리오에 대해 E2E 테스트를 작성하고 실행해줘.
실패하는 테스트가 있으면 코드를 수정해서 통과시켜줘.
최소 50개 시나리오를 커버해야 해.
```

### 스모크 테스트 프롬프트
```
backend에 스모크 테스트를 추가해줘.
서버 시작, 헬스체크, 회원가입, 로그인이 동작하는지 확인하는 최소한의 테스트야.
커밋 전에 항상 이 테스트를 먼저 통과해야 해.
```

### 버그 수정 프롬프트
```
다음 버그들을 수정하고 각각에 대한 회귀 테스트를 추가해줘:
1. 강의실 초대코드가 화면에 표시되지 않음
2. /admin/kpi 페이지가 404
3. Router 관련 콘솔 경고
```

프롬프트에서 핵심은 **"테스트가 실패하면 코드를 수정해서 통과시켜줘"**라는 지시입니다. 이렇게 하면 AI가 테스트 작성과 버그 수정을 동시에 진행합니다.

## 배운 점

### 1. 구현 직후의 테스트가 가장 중요하다
코드를 작성한 직후가 버그를 가장 쉽게 발견할 수 있는 시점입니다. 시간이 지나면 코드의 맥락을 잊어버리기 때문에 디버깅이 훨씬 어려워집니다.

### 2. 스모크 테스트는 "안전벨트"다
모든 변경 전후에 스모크 테스트를 실행하면, 기본적인 기능이 깨졌을 때 즉시 알 수 있습니다. 30초면 실행되므로 부담도 없습니다.

### 3. 버그 하나당 테스트 하나
버그를 수정할 때마다 그 버그를 재현하는 테스트를 작성하세요. 이것이 **회귀 테스트**입니다. 이렇게 하면 같은 버그가 다시 발생했을 때 자동으로 감지됩니다.

### 4. E2E 테스트의 가치
단위 테스트만으로는 "전체 흐름"이 동작하는지 알 수 없습니다. 사용자가 실제로 하는 행동(가입 → 로그인 → 회사 설립 → 과제 제출)을 시뮬레이션하는 E2E 테스트가 있어야 사용자 경험 전체를 검증할 수 있습니다.

### 5. AI 생성 코드도 테스트가 필수다
AI가 작성한 코드라고 해서 버그가 없는 것은 절대 아닙니다. 오히려 AI는 문맥을 놓치거나, API 간 연결을 잘못하는 경우가 있습니다. **AI가 만든 코드일수록 꼼꼼한 테스트가 필요합니다.**

### 6. 디버깅은 탐정 작업이다
"댓글이 안 보여요" 같은 증상만으로는 원인을 알 수 없습니다. 데이터가 저장은 되는지? API가 응답을 하는지? SQL이 올바른지? 하나씩 확인하며 범위를 좁혀가는 것이 체계적인 디버깅입니다.
