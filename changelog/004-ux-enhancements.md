## 무엇을 했나요?

MVP의 기본 기능이 동작하는 것을 확인한 후, 사용자 경험(UX)을 크게 개선했습니다:

- **마크다운 위지윅 에디터 도입**: 글 작성 시 서식을 시각적으로 확인하며 편집
- **파일 업로드 기능**: 이미지, 문서 등을 첨부할 수 있도록 추가
- **더보기 토글**: 긴 글을 접어서 보여주는 UI
- **외주 작업완료 보고서**: 마켓에서 외주 거래 후 보고서 작성 기능
- **마켓 자동 포스팅**: 외주 완료 시 마켓에 자동으로 결과물 게시
- **SQLite 데드락 수정**: 자동 포스팅 중 발생한 동시성 문제 해결
- **전체 입력 필드 에디터 전환**: 모든 상세 내용 입력란을 마크다운 에디터로 통일
- **태그 입력 UI + 클릭 필터링**: 게시물에 태그를 달고 태그로 필터링하는 기능

## 왜 필요했나요?

### 기능이 있는 것과 "쓰고 싶은" 것은 다르다

MVP가 동작한다고 해서 사용자가 만족하는 것은 아닙니다. 예를 들어:

```
일반 텍스트 입력                마크다운 에디터
─────────────────            ─────────────────
그냥 텍스트만 입력 가능          볼드, 이탤릭, 제목 지원
줄바꿈도 이상하게 보임           미리보기 제공
코드를 넣으면 읽기 어려움         코드 하이라이팅
이미지 첨부 불가                드래그앤드롭 이미지 삽입
```

학생들이 과제 보고서, 회사 소개, 마켓 상품 설명을 작성할 때 서식이 지원되지 않으면 표현력이 크게 제한됩니다.

### 태그가 왜 필요한가?

마켓에 상품이 100개가 되면 원하는 것을 찾기 어렵습니다:

```
태그 없이                     태그 있으면
─────────                   ──────────
모든 상품이 한 줄로 나열         #디자인 #개발 #마케팅
스크롤해서 하나씩 확인          원하는 태그 클릭으로 필터링
검색으로만 찾을 수 있음          관련 상품을 한눈에 확인
```

### 외주 자동 포스팅이 필요한 이유

마켓에서 외주(아웃소싱) 거래가 완료되면, 그 결과물을 다른 학생들도 볼 수 있어야 학습 효과가 극대화됩니다. 수동으로 포스팅하면 잊어버리거나 귀찮아서 안 하게 되므로, 자동화가 필요합니다.

## 어떻게 만들었나요?

### 마크다운 위지윅 에디터

**마크다운(Markdown)**은 간단한 기호로 서식을 표현하는 방법입니다:

```markdown
# 제목 1
## 제목 2
**볼드**
*이탤릭*
- 목록 항목
```

**위지윅(WYSIWYG: What You See Is What You Get)**은 편집하면서 최종 결과를 바로 볼 수 있는 에디터입니다. 워드프로세서처럼 버튼을 클릭하면 서식이 적용됩니다.

이 프로젝트에서는 둘을 결합한 **마크다운 위지윅 에디터**를 도입했습니다:

```
┌──────────────────────────────────────────┐
│ B  I  H1  H2  📎  🖼️  </>  링크  │ ← 도구 모음
├──────────────────────────────────────────┤
│                                          │
│  여기에 글을 작성하면                       │
│  실시간으로 서식이 적용되어 보입니다          │
│                                          │
│  ## 이런 제목도 바로 보이고                 │
│  **볼드**도 즉시 적용됩니다                 │
│                                          │
└──────────────────────────────────────────┘
```

기술적으로는 React용 마크다운 에디터 라이브러리를 사용했습니다:

```tsx
// 마크다운 에디터 컴포넌트 사용 예시
import { MarkdownEditor } from '@/components/MarkdownEditor';

function AssignmentSubmit() {
  const [content, setContent] = useState('');

  return (
    <MarkdownEditor
      value={content}
      onChange={setContent}
      placeholder="과제 내용을 작성하세요..."
    />
  );
}
```

### 파일 업로드 구현

파일 업로드는 프론트엔드와 백엔드가 협력해야 하는 기능입니다:

```
사용자가 파일 선택 (또는 드래그앤드롭)
    ↓
프론트엔드: FormData로 파일을 감싸서 전송
    ↓
백엔드: 파일을 서버 디스크에 저장
    ↓
백엔드: 저장된 파일의 URL을 응답
    ↓
프론트엔드: 에디터에 이미지/파일 링크 삽입
```

Go 백엔드에서의 파일 처리:

```go
// 파일 업로드 핸들러의 핵심 로직
func UploadFile(c echo.Context) error {
    // 1. 요청에서 파일 추출
    file, err := c.FormFile("file")
    if err != nil {
        return c.JSON(400, map[string]string{"error": "파일이 없습니다"})
    }

    // 2. 파일 크기 제한 확인 (10MB)
    if file.Size > 10*1024*1024 {
        return c.JSON(400, map[string]string{"error": "파일이 너무 큽니다"})
    }

    // 3. 안전한 파일명 생성 (UUID + 원본 확장자)
    ext := filepath.Ext(file.Filename)
    safeName := uuid.New().String() + ext

    // 4. 디스크에 저장
    dst, _ := os.Create("uploads/" + safeName)
    src, _ := file.Open()
    io.Copy(dst, src)

    // 5. URL 반환
    return c.JSON(200, map[string]string{
        "url": "/uploads/" + safeName,
    })
}
```

파일 업로드에서 주의할 보안 사항:
- 원본 파일명을 그대로 사용하면 안 됨 (경로 조작 공격 가능)
- 파일 크기를 제한해야 함 (서버 디스크 가득 참 방지)
- 실행 가능한 파일(.exe, .sh)은 차단해야 함

### 더보기 토글

긴 글을 처음부터 전부 보여주면 페이지가 너무 길어집니다:

```
더보기 토글 적용 전            더보기 토글 적용 후
──────────────────          ──────────────────
과제 1: (전체 내용 500줄)     과제 1: (처음 3줄)
과제 2: (전체 내용 300줄)              [더보기 ▼]
과제 3: (전체 내용 400줄)     과제 2: (처음 3줄)
→ 스크롤이 끝나지 않음              [더보기 ▼]
                            → 깔끔하고 필요한 것만 펼쳐봄
```

React에서 더보기 토글 구현:

```tsx
function ExpandableContent({ content, maxLines = 3 }) {
  const [expanded, setExpanded] = useState(false);

  return (
    <div>
      <div className={expanded ? '' : 'line-clamp-3'}>
        {content}
      </div>
      <button onClick={() => setExpanded(!expanded)}>
        {expanded ? '접기 ▲' : '더보기 ▼'}
      </button>
    </div>
  );
}
```

`line-clamp-3`은 Tailwind CSS의 클래스로, 텍스트를 3줄까지만 보여주고 나머지는 잘라냅니다.

### 태그 시스템

태그 시스템은 **입력**과 **필터링** 두 부분으로 구성됩니다:

**태그 입력 UI:**

```
┌──────────────────────────────────────┐
│ 태그: [디자인] [React] [입력...]  ×   │
└──────────────────────────────────────┘

- 텍스트 입력 후 Enter 또는 쉼표로 태그 추가
- 태그 옆 × 버튼으로 삭제
- 중복 태그 방지
```

**태그 필터링:**

```
전체 상품 목록 상단:
[전체] [#디자인] [#개발] [#마케팅] [#기획] [#데이터]
         ↓ 클릭
해당 태그가 달린 상품만 표시
```

데이터베이스에서 태그 저장 방식:

```sql
-- 태그 테이블
CREATE TABLE tags (
    id INTEGER PRIMARY KEY,
    name TEXT UNIQUE NOT NULL
);

-- 게시물-태그 연결 테이블 (다대다 관계)
CREATE TABLE post_tags (
    post_id INTEGER REFERENCES posts(id),
    tag_id INTEGER REFERENCES tags(id),
    PRIMARY KEY (post_id, tag_id)
);
```

이 구조를 **다대다(Many-to-Many) 관계**라고 합니다:
- 하나의 게시물에 여러 태그를 달 수 있음
- 하나의 태그가 여러 게시물에 사용될 수 있음

### SQLite 데드락 수정

자동 포스팅 기능을 추가하면서 **데드락(Deadlock)** 문제가 발생했습니다.

데드락이란 두 작업이 서로의 자원을 기다리며 영원히 멈추는 상태입니다:

```
작업 A: 테이블1 잠금 → 테이블2 필요 (기다림)
작업 B: 테이블2 잠금 → 테이블1 필요 (기다림)
→ 둘 다 영원히 기다림 (데드락!)
```

SQLite에서는 **쓰기 작업이 동시에 하나만 가능**하기 때문에 이런 문제가 더 자주 발생합니다.

해결 방법:
```go
// 수정 전: 각각 독립적인 트랜잭션
db.Exec("UPDATE market_items SET status = 'completed' ...")
db.Exec("INSERT INTO feed_posts ...")  // 여기서 데드락!

// 수정 후: 하나의 트랜잭션으로 묶기
tx, _ := db.Begin()
tx.Exec("UPDATE market_items SET status = 'completed' ...")
tx.Exec("INSERT INTO feed_posts ...")
tx.Commit()
```

트랜잭션으로 묶으면 두 작업이 **원자적(atomic)**으로 실행됩니다. 둘 다 성공하거나, 둘 다 실패합니다.

## 사용한 프롬프트

### 마크다운 에디터 프롬프트
```
모든 상세 내용 입력 필드에 마크다운 위지윅 에디터를 적용해줘.
파일 업로드(드래그앤드롭)도 지원하고, 긴 글은 더보기 토글로 접어서 보여줘.
```

### 태그 기능 프롬프트
```
마켓과 피드에 태그 기능을 추가해줘.
입력 시 자동완성, 클릭으로 필터링, 태그별 색상 구분까지.
리그레션 테스트도 함께 작성해줘.
```

### 데드락 수정 프롬프트
```
외주 자동 포스팅에서 SQLite 데드락이 발생해.
동시에 market_items 업데이트와 feed_posts 삽입이 일어나는 부분을 수정해줘.
```

프롬프트에서 핵심은 **기존 기능과의 연결**을 명시하는 것입니다. "에디터 추가해줘"보다 "모든 상세 내용 입력 필드에 적용해줘"라고 하면 누락 없이 적용됩니다.

## 배운 점

### 1. UX는 "있으면 좋은 것"이 아니다
사용자가 제품을 쓰고 싶어야 서비스가 성공합니다. 아무리 좋은 기능이어도 사용하기 불편하면 아무도 쓰지 않습니다. 스타트업에서 UX 개선은 기능 추가만큼 중요합니다.

### 2. 동시성 문제는 반드시 발생한다
여러 사용자가 동시에 같은 데이터에 접근하면 예상치 못한 문제가 발생합니다. 특히 SQLite처럼 쓰기 잠금이 있는 DB에서는 트랜잭션 관리가 매우 중요합니다.

### 3. 작은 UI 개선이 큰 차이를 만든다
더보기 토글, 태그 필터링 같은 작은 기능이 전체 사용 경험을 크게 개선합니다. 이런 디테일에 신경 쓰는 것이 좋은 제품의 조건입니다.

### 4. 리그레션 테스트의 가치
새 기능을 추가할 때마다 기존 기능이 깨지지 않는지 자동으로 확인하는 리그레션 테스트가 있으면, 안심하고 새 기능을 추가할 수 있습니다.

### 5. 일관성이 사용성을 높인다
모든 입력 필드를 동일한 마크다운 에디터로 통일하면, 사용자는 한 번만 배우면 모든 곳에서 같은 방식으로 사용할 수 있습니다. 화면마다 다른 에디터를 쓰면 혼란스럽습니다.

### 6. 자동화는 행동을 유도한다
"외주 완료 후 결과물을 포스팅하세요"보다 자동으로 포스팅되도록 하면 참여율이 훨씬 높아집니다. 사용자에게 행동을 요구하는 대신 시스템이 자동으로 처리하는 것이 더 효과적입니다.
