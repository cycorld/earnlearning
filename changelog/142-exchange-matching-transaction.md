# 142. 거래소 체결을 통째로 트랜잭션으로 묶기 ("전부 성공 아니면 전부 취소")

**날짜**: 2026-06-30
**티켓**: #142
**타입**: 버그수정 (데이터 무결성)

## 무엇을 했나요?

거래소에서 매수·매도가 체결될 때 일어나는 **여러 단계의 DB 쓰기**(체결 기록 → 주문 상태 갱신 → 지갑 차감/입금 → 주식 소유권 이전 → 회사 가치 갱신)를 **하나의 트랜잭션**으로 묶었습니다. 이제 중간 어느 단계가 실패하면 **앞 단계까지 전부 자동 취소(롤백)** 됩니다.

## 왜 필요했나요?

#140 에서 체결 중 발생하던 패닉(서버가 죽던 버그)을 고쳤습니다. 하지만 그건 "패닉이라는 한 가지 실패 경로"만 막은 것이었습니다. `runMatching` 은 여전히 트랜잭션 없이 여러 쓰기를 순서대로 실행하고 있었고, **패닉이 아닌 다른 실패**(DB 에러, 잔액 부족, 제약 위반 등)가 중간에 나면 **앞 단계만 저장된 반쪽 상태**가 그대로 남았습니다.

예를 들어 ① 체결 기록 + ② 매수자 지갑 차감 + ③ 매도자 지갑 입금까지 저장된 뒤 ④ 주식 이전에서 실패하면 → **돈은 움직였는데 주식은 안 넘어간** 상태가 영구히 남습니다. #140 의 "배운 점"에서 후속 과제로 명시해 둔 바로 그 작업입니다.

## 어떻게 만들었나요?

### 1) `DBTX` 인터페이스 — DB와 트랜잭션을 같은 얼굴로

`*sql.DB`(연결 풀)와 `*sql.Tx`(트랜잭션)는 둘 다 `Exec`/`Query`/`QueryRow` 를 가집니다. 이 셋만 요구하는 작은 인터페이스 `DBTX` 를 만들면, **저장소(repository) 코드는 자기가 일반 연결 위에서 도는지 트랜잭션 안에서 도는지 몰라도** 똑같이 동작합니다.

```go
type DBTX interface {
    Exec(query string, args ...any) (sql.Result, error)
    Query(query string, args ...any) (*sql.Rows, error)
    QueryRow(query string, args ...any) *sql.Row
}
```

각 저장소에 `WithTx(tx)` 메서드를 추가해, **같은 트랜잭션에 묶인 복제본**을 만들 수 있게 했습니다. 체결 처리는 시작할 때 `tx := db.Begin()` 한 뒤 `exchangeRepo.WithTx(tx)`, `walletRepo.WithTx(tx)`, `companyRepo.WithTx(tx)` 로 모든 쓰기를 한 트랜잭션에 모읍니다.

### 2) 외부 부수효과는 **커밋 후로** 분리

알림(`notify`)과 거래소 채널 자동 글쓰기(`autoPoster`)는 트랜잭션 **안에서** 실행하면 안 됩니다. 두 가지 이유:

- **롤백과 엉킴**: 트랜잭션이 취소됐는데 "체결됐어요" 알림이 이미 나가면 거짓 알림이 됩니다.
- **데드락**: 이 프로젝트의 SQLite 는 `SetMaxOpenConns(1)`(연결 1개)로 운영됩니다. 트랜잭션이 그 하나뿐인 연결을 쥐고 있는 동안, 알림 코드가 **다른 연결**로 DB 에 접근하려 하면 영원히 기다리는 **교착 상태**가 됩니다.

그래서 체결마다 알림/자동글 정보를 `tradeNotice` 로 모아 두었다가, **`tx.Commit()` 이 성공한 뒤에만** 한꺼번에 발송합니다.

### 3) 주주 이전 로직 단순화 (덤)

기존엔 매수자 분기에서 `FindShareholder` → (없으면 생성 / 있으면 수정)으로 갈라지고, 이를 위해 별도 컴포넌트 `ShareholderUpdater` 를 두고 있었습니다. 이미 `CompanyRepo` 에 있던

- `UpsertShareholder(...)` — 없으면 추가, 있으면 더해줌(매수자에 딱 맞음)
- `SubtractShareholderShares(...)` — 빼주고 0 이면 행 삭제(매도자에 딱 맞음)

로 교체했습니다. 분기·nil 체크가 사라져 코드가 짧아지고, #140 식 nil 역참조 위험도 원천 제거됩니다. 더는 쓰이지 않게 된 `ShareholderUpdater` 와 구현 파일은 제거했습니다.

### TDD (회귀 테스트)

`exchange_atomic_test.go`: 체결 도중 **매도자 지갑 입금(Credit) 단계에서 일부러 실패**를 주입(`failingWalletRepo`)하고 매수 주문을 체결시킵니다. 그 시점엔 이미 체결 기록·주문 갱신·매수자 차감이 트랜잭션 안에서 실행된 상태입니다. 수정 후엔:

- 체결 기록(`stock_trades`) 0건 — 남지 않음
- 매수자 잔액 그대로 — 차감이 롤백됨
- 매수자 주주 레코드 없음 — 주식 이전이 롤백됨
- 매도자 주식 그대로 — 차감이 롤백됨

전부 "없던 일"이 됐는지 검증합니다. 트랜잭션이 없던 옛 코드라면 체결 기록과 매수자 차감이 남아 이 테스트는 실패합니다.

## 배운 점

- **"여러 번의 쓰기"는 곧 "반쪽 실패의 위험"이다.** 한 줄 한 줄은 멀쩡해도, 중간에 멈추면 데이터가 모순 상태로 남습니다. 돈·소유권처럼 보존돼야 하는 값을 다룰 땐 **전체를 트랜잭션으로 묶는 것**이 기본기입니다.
- **트랜잭션 안에서는 "그 트랜잭션"으로만 DB에 접근해야 한다.** 연결이 1개뿐인 SQLite 에선, 트랜잭션 도중 다른 연결로 쿼리하면 교착에 빠집니다. 그래서 저장소를 `WithTx` 로 트랜잭션에 다시 묶고, 외부 부수효과(알림 등)는 커밋 뒤로 미뤘습니다.
- **인터페이스 하나(`DBTX`)로 "연결이냐 트랜잭션이냐"를 감추면** 저장소 메서드를 두 벌로 만들 필요가 없습니다. 같은 코드가 단독 호출에서도, 더 큰 트랜잭션의 일부로도 동작합니다.
- 이 수정은 **앞으로의 손상**을 막습니다. #140 이전에 이미 생겼을 **과거 손상분**(돈만 이동·주식 미이전) 점검·보정은 후속 #143 에서 다룹니다.
