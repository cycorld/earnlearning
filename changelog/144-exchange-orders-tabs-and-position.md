# 144. 거래소 주문 UX 고도화 — 내 주문 상태 탭 + 보유/여유자금 표시·제약

**날짜**: 2026-06-30
**티켓**: #144
**타입**: 기능추가 / UX

## 무엇을 했나요?

거래소 주문 경험을 두 군데에서 손봤습니다.

### A. 거래소 메인 "내 주문" → 상태 탭
- `[진행중] [체결] [취소] [전체]` 탭으로 분리 (각 탭에 개수 배지). 기본은 진행중.
- 행 정보 보강: **실제 회사명·로고**(기존엔 항상 "기업 #67" 처럼 떴음), 매수/매도, 수량×가격, 상태 한글(대기/부분체결/체결/취소), 부분체결 시 잔여 수량, 경과 시간.
- 진행중(대기·부분체결) 주문은 목록에서 **바로 취소** (X 버튼).

### B. 주문 폼에 보유 주식 + 여유자금 표시 + 입력 제약
- 종목 상세 주문 패널에 **"보유 N주 (매도가능 M)"** 와 **"여유자금 ₩X"** 를 항상 표시.
- **매수는 여유자금 한도, 매도는 보유(매도가능) 한도** 안에서만 주문 가능. 초과하면 빨간 경고 + 주문 버튼 비활성.
- 수량 옆 **"최대"** 버튼: 매수는 여유자금÷가격, 매도는 매도가능 수량으로 한 번에 채움.

## 왜 필요했나요?

- "내 주문"이 상태 구분 없이 한 덩어리라 진행중인 주문과 끝난 주문이 섞여 보기 불편했습니다. 회사명도 안 떠서 어느 종목 주문인지 알기 어려웠습니다.
- 주문할 때 **내가 이 회사 주식을 몇 주 갖고 있는지, 돈이 얼마 남았는지** 화면에 안 보여서, 매도하려는데 주식이 모자라거나 매수하려는데 잔액이 부족한 걸 **주문 버튼을 눌러 서버 에러가 나야** 알 수 있었습니다.

## 어떻게 만들었나요?

### 백엔드 — 포지션 조회 API 신설
- `GET /exchange/position/:companyId` → `{ shares, available_shares, balance, available_cash }`
  - `available_shares` = 보유 − **미체결 매도**(pendingSell), `available_cash` = 잔액 − **미체결 매수**(pendingBuy)
  - **중요**: 이 값들은 주문을 실제로 검증하는 `PlaceOrder` 가 쓰는 값과 **똑같습니다.** 그래서 프론트 제약 = 백엔드 제약 — 화면에서 막히는 기준과 서버가 거부하는 기준이 어긋나지 않습니다.
- 계산에 쓰는 repo 함수(`FindShareholder`, `GetPendingBuyTotal`, `GetPendingSellShares`, 지갑 조회)는 전부 `PlaceOrder` 가 이미 쓰던 것 — 재사용.
- TDD: 창업자(10000주 보유) → 미체결 매도 5주 → `available_shares` 9995, 미체결 매수 3주×8000 → `available_cash` 가 24,000원 줄어드는지 검증.

### 프론트
- **메인(ExchangePage)**: 주문은 회사명을 안 주므로 `/exchange/companies` 응답으로 `company_id → 회사` 맵을 만들어 이름·로고를 채움. 탭은 클라이언트에서 상태로 필터(개수도 함께). 주문 전량을 위해 `?limit=100` 으로 조회.
- **상세(ExchangeDetailPage)**: 포지션을 같이 불러와 표시. 제약은 프론트(버튼 비활성 + 인라인 경고 + 제출 직전 재확인)와 백엔드 양쪽에서 검증.

### 검증
- 백엔드 스모크 24/24 + 거래소 테스트(포지션·교차체결) 통과.
- 프론트 빌드/타입체크 통과, 테스트 208/208.

## 배운 점

- **"화면 제약"과 "서버 제약"은 같은 값을 봐야 한다.** 프론트가 자기 식으로 한도를 계산하면(예: 미체결 주문을 모르고 잔액 전체를 한도로 잡으면) 서버는 거부하는데 화면은 통과시키는 어긋남이 생깁니다. 그래서 서버 검증이 쓰는 값을 그대로 내려주는 API를 따로 뒀습니다.
- **미체결 주문도 "묶인 돈/주식"이다.** 여유자금 = 잔액이 아니라 잔액 − 이미 걸어둔 매수금액. 매도가능 = 보유가 아니라 보유 − 이미 걸어둔 매도수량.
