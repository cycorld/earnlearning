# 잘못 올라간 글, 관리자가 옮겨주기 — 게시물 카테고리 수정

## 무엇을 했나요?

관리자가 게시물을 수정할 때 **카테고리(채널)도 변경**할 수 있게 했습니다. 학생이 자유게시판에 올릴 글을 과제 채널에 올렸다면, 이제 관리자가 수정 창에서 채널을 골라 옮겨줄 수 있습니다.

## 왜 필요했나요?

관리자는 이미 학생 게시물의 본문과 태그를 고칠 수 있었습니다. 그런데 정작 가장 흔한 실수 — **채널을 잘못 고른 글** — 은 옮길 방법이 없었습니다. 백엔드의 수정 입력(`UpdatePostInput`)에 `channel_id` 필드 자체가 없었기 때문입니다. 삭제하고 다시 올리라고 하면 좋아요·댓글이 다 날아가니, "옮기기"가 필요했습니다.

## 어떻게 만들었나요? (TDD)

**테스트 먼저** 작성했습니다 (Red → Green):

1. 관리자가 같은 강의실의 다른 채널로 옮기면 성공해야 한다
2. **일반 작성자**가 자기 글의 채널을 바꾸려 하면 거부되어야 한다 (권한 경계)
3. 존재하지 않는 채널로는 못 옮긴다 (존재 검증)
4. **다른 강의실**의 채널로는 못 옮긴다 (classroom 경계 — 강의실 격리 원칙 #159)
5. `channel_id`를 안 보내면 기존 채널이 유지된다 (하위 호환)

그 다음 최소한의 코드로 통과시켰습니다:

- `UpdatePostInput`에 `ChannelID *int` 추가 — **포인터**를 쓴 이유: JSON에서 "안 보냄"과 "0을 보냄"을 구분하기 위해서입니다. `nil`이면 채널 유지.
- usecase에서 검증 3종: 관리자인가? → 새 채널이 존재하는가? → 기존 채널과 같은 강의실인가?
- repo에 `UpdatePostChannel` 메서드 추가 (기존 `UpdatePost` 시그니처는 건드리지 않음 — 최소 변경).

프론트엔드는 FeedPage의 기존 수정 다이얼로그에 **관리자에게만 보이는** 카테고리 select를 추가했습니다. 채널 목록은 피드 탭에서 이미 불러온 `channels` 상태를 재사용하고, 저장 성공 시 목록의 해당 글 채널 뱃지도 즉시 갱신합니다.

## 배운 점

- **"없는 필드"는 조용한 제약입니다.** API가 거부하는 게 아니라 무시하는 것이라, 프론트에서 보내도 티가 안 납니다. 입력 구조체에 필드를 추가할 때는 반드시 서버 검증도 함께 설계해야 합니다.
- Go에서 `*int` 포인터 필드는 "값이 없음"을 표현하는 관용구입니다. `0`이 유효한 값일 수 있는 ID 필드에서 특히 중요합니다.
- 권한 검사는 "누가"(admin)만이 아니라 "어디까지"(같은 classroom)도 함께 봐야 합니다. 강의실 격리가 있는 시스템에서는 모든 이동/참조 기능에 경계 테스트가 필요합니다.
