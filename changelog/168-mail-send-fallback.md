# 메일이 502로 안 나가던 밤 — IAM 권한과 폴백 이야기

## 무엇을 했나요?

메일함(#166)을 프로덕션에 배포하고 실제 메일 왕복 테스트를 하는데, 발송이 `502`로 죽었습니다. 원인을 찾아 폴백을 고쳤습니다.

## 왜 그랬나요?

프로덕션의 AWS 자격(IAM 유저)은 보안상 `noreply@earnlearning.com` **한 주소로만** 발송을 허용하게 잠겨 있었습니다. 학생 주소(`kim@earnlearning.com`)를 From으로 쓰려 하자 AWS가 `AccessDeniedException`으로 거부했는데, 우리 폴백 코드는 "미인증 주소" 거부 문구만 알아듣고 "권한 없음" 거부는 몰랐습니다. 그래서 폴백(설정 From + Reply-To=학생주소)이 발동하지 못하고 그대로 502.

발송 실패가 로그에도 안 남아서, EC2에서 aws cli로 같은 호출을 재현하고서야 진짜 에러 메시지를 봤습니다.

## 어떻게 고쳤나요?

- 폴백 판정을 "이 From 신원을 못 쓴다"로 넓혔습니다: 미인증 + **AccessDenied** 둘 다 → `noreply@` From + `Reply-To`=학생주소로 재시도. 받는 사람이 답장하면 학생 주소로 갑니다.
- 발송 실패를 이제 로그에 남깁니다. 다음엔 aws cli 재현 없이 로그만 보면 됩니다.
- 회귀 테스트에 **prod 실제 에러 메시지**를 박아뒀습니다.

## 배운 점

- 폴백 코드는 "어떤 에러 문구가 오는가"에 의존합니다. 실제 운영에서 오는 에러는 상상과 다를 수 있으니, **실제 메시지를 테스트에 보존**하는 게 최고의 문서입니다.
- 실패가 사용자에게 5xx로만 보이고 서버 로그에 안 남으면 디버깅 비용이 몇 배로 뜁니다. 실패 경로에는 반드시 로그.
