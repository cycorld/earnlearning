# 청산 안건 UX 개선 — 세금 20% 안내 + 가결 즉시 자동 집행

## 왜 필요했는가

#023에서 회사 청산 기능을 처음 만들었을 때, 절차는 이렇게 설계됐어요:

1. 주주가 "회사 청산" 안건을 상정한다
2. 주주들이 투표한다 (기본 기준 70% 찬성)
3. 가결되면 대표가 "청산 집행" 버튼을 누른다
4. 자산이 주주들에게 분배된다

그런데 이걸 실제로 학생들에게 풀어놓고 보니 두 가지 **구멍**이 생겼습니다.

### 문제 1. 세금 20%를 알리지 않고 투표하게 된다

청산 시에는 회사 자산의 20%를 세금(시스템/정부)으로 낸 뒤 남은 80%만 주주들에게 분배합니다. 그런데 청산 안건을 상정할 때 이 사실을 **아무도 알려주지 않아요**. 주주들은 "전부 돌려받는 줄 알고" 투표했다가, 집행 결과를 보고 "어? 왜 20% 깎였지?" 라는 당황스러운 순간을 맞이합니다.

현실 세계에선 청산 공시 문서에 세금/비용이 기재되지만, 우리 LMS는 안건 본문이 곧 모든 것이었죠. **의사결정자(주주)가 정보 없이 투표하는 상황**은 게임 디자인상 문제입니다.

### 문제 2. 가결됐는데 "집행"이 누락된다

투표가 통과돼서 `passed` 상태가 되면, 거기서 플로우가 **한 번 끊깁니다**. 대표가 회사 페이지에 들어가 "청산 집행" 버튼을 눌러야 진짜 자산이 이동해요.

그런데 대표가 그 버튼을 잊거나, 타이밍을 놓치거나, 심지어 "가결됐으면 끝난 거 아냐?"라고 오해하면 → **주주 자산이 회사 지갑에 잠긴 채로 방치**됩니다. 이화여자대학교 한 팀은 실제로 이렇게 며칠 넘게 자산이 묶여 있었어요.

사용자가 직접 제안을 줬습니다:

> "앞으로 청산 투표 올릴 때, 세금 20% 공지하고, 청산 투표 통과하면 알아서 분배되도록 개선해줘."

정확한 진단입니다. 두 가지를 한 번에 고칩니다.

## 무엇을 했는가

### 1. 청산 안건 설명에 세금 안내 자동 prepend

`CreateProposal` 호출 시 `proposal_type === 'liquidation'` 이면 본문 맨 앞에 아래 문구를 자동으로 붙입니다:

```
⚠️ **청산 안내**: 가결 시 회사 자산의 **20%는 세금**으로 납부되고,
나머지 **80%가 주주별 지분율에 따라 자동 분배**됩니다.
가결 즉시 집행되며 회사는 영구 정지됩니다.

---
```

이렇게 하면:
- 안건을 상정한 주주가 까먹고 안 적어도 **항상** 세금 공지가 있습니다
- 코드에서 한 곳(`liquidationTaxNotice` 상수)을 바꾸면 모든 기존/신규 안건에 자동 반영
- 사용자가 추가로 적은 설명은 그 뒤에 그대로 보존됩니다

### 2. 가결 즉시 자동 집행

`closeProposal` 함수 — 투표 집계 후 proposal이 `passed` 상태가 되면 호출되는 내부 함수 — 에 로직을 하나 끼워넣었습니다:

```go
if finalStatus == company.ProposalStatusPassed &&
    p.ProposalType == company.ProposalTypeLiquidation &&
    c.Status == "active" {
    if _, err := uc.executeLiquidationCore(p, c, p.ProposerID); err != nil {
        // 실패해도 passed 상태는 유지 — 수동 /execute 로 재시도 가능
        log.Printf("...")
    } else {
        // 성공: 이미 per-shareholder 분배금 알림이 나가므로
        // 일반 "주주총회 결과" 알림은 생략
        return p, nil
    }
}
```

이를 위해 기존 `ExecuteLiquidation` 을 두 층으로 쪼갰어요:

- `ExecuteLiquidation(proposalID, userID)` — 외부 API 엔드포인트용. 권한 체크 + 상태 체크 후 아래 core 호출
- `executeLiquidationCore(p, c, actorID)` — 순수 분배 로직. 권한 체크 없음. 자동 집행 경로가 이걸 직접 부름

이 구조 덕에:
- 기존 수동 `/execute` API는 그대로 작동 (fallback)
- 자동 집행 실패 시 passed 상태 유지 → 대표가 /execute로 재시도 가능
- 권한 체크가 한 번만 있으므로 우회 구멍 없음

### 3. 권한 체크 순서 보정

부수적 개선: `ExecuteLiquidation`의 체크 순서를 이렇게 바꿨습니다.

**Before**: type → status → dissolved → shareholder
**After**: type → **shareholder** → status → dissolved

자동 집행이 도입되면 대부분 청산은 이미 `executed/dissolved`로 들어가 있을 확률이 높습니다. 이때 non-shareholder가 `/execute`를 때리면 예전 코드에선 "이미 청산됨" 에러가 떠서 **왜 본인이 거부됐는지 모호**했어요. 이제 "당신은 주주가 아닙니다" 에러가 먼저 뜨므로 명확합니다.

### 4. 프론트엔드 — 안건 상정 폼에 세금 배너

`ProposalSection.tsx`에서 안건 종류로 "회사 청산"을 선택하면 폼에 경고 배너가 나타납니다:

```
⚠️ 청산 안내
• 가결 시 회사 자산의 20%는 세금으로 납부됩니다.
• 나머지 80%가 주주별 지분율에 따라 자동 분배됩니다.
• 가결 즉시 집행되며, 회사는 영구 정지됩니다.
```

백엔드에서 본문 prefix로도 안내하지만, 안건을 **작성하는 순간**에도 보여주면 작성자가 더 신중하게 제목/설명을 쓸 수 있죠.

### 5. 회귀 테스트 2종 + 기존 테스트 업데이트

새로 추가한 테스트:

- `TestLiquidation_CreateProposal_IncludesTaxNotice`: 청산 안건 본문에 "세금" + "20%" 문자열이 자동으로 들어가는지, 또 사용자가 입력한 설명이 보존되는지 검증
- `TestLiquidation_AutoExecute_OnProposalPass`: 100% 주주가 yes 투표만 하면 `/execute` 호출 없이도 proposal.status='executed', company.status='dissolved', 지갑에 분배금(8M = 10M - 20%)이 들어왔는지 확인

기존 청산 테스트 5개는 "투표 → 수동 /execute" 패턴이었는데, 이제 자동 집행이 되므로 `/execute` 호출 단계를 제거하고 최종 상태를 직접 검증하는 방식으로 바꿨습니다. `TestLiquidation_DoubleExecute_Rejected`는 `TestLiquidation_ManualExecuteAfterAutoExecute_Rejected`로 의미를 보존하며 리네임했고, `TestProposal_ThresholdReached_Passes`도 threshold 로직 단독 검증을 위해 general 타입으로 바꿨어요.

전체 통합 테스트 264개 모두 통과.

## 배운 점

### 게임 룰은 UI에 선언되어야 한다

"가결되면 20% 세금"은 백엔드 상수 `LiquidationTaxPercent = 20`에 박혀 있었지만, **주주들에겐 단 한 번도 표시되지 않고 있었어요**. 정책 상수는 코드가 아니라 **사용자가 보는 곳**에 나와야 의미가 있습니다. 특히 "내 자산 처분"을 결정하는 규칙이라면 더더욱.

### "집행" 단계는 자동화 대상 1순위

투표는 의사결정이고, 분배는 **그 의사결정의 기계적 귀결**입니다. 사람이 사이에 끼면:
- 잊어버릴 수 있음 (실제로 발생)
- 타이밍이 조정 권한이 돼서 **권력 남용** 가능 (대표가 "내가 집행 버튼 안 누르면 분배 안 됨" 식의 지연 전략)
- 학생이 "이해를 못해서" 막힐 수 있음

자동화 가능한 건 자동화하고, 사람 판단이 필요한 지점(투표)에만 인간을 둔다 — 이게 플로우 설계의 기본입니다.

다만 실패 대비는 남겨둬야 해요:
- 자동 집행이 실패해도 `passed` 상태는 유지
- 대표가 수동 `/execute`로 재시도 가능
- 수동 엔드포인트는 fallback으로 살아있음

"자동화하되, 탈출구를 남긴다" 는 원칙.

### 내부 공통 함수를 뽑는 타이밍

`ExecuteLiquidation`을 두 층으로 쪼갠 순간은, **"같은 로직을 두 지점에서 호출해야 할 때"** 였습니다. 더 일찍(처음부터) 분리했으면 좋았을까? 아마 아닐 거예요. 당시엔 수동 API 하나뿐이었으니 과잉 설계였을 겁니다. **실제 두 번째 호출자가 생긴 지금이 쪼개기 딱 좋은 때**입니다.

규칙화하면: "한 번 쓰는 private 함수는 만들지 마라, 두 번 쓰게 되면 그 때 뽑아라."
