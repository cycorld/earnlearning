# 034. OAuth API 접근 버그 수정 + 스코프 기반 권한 제한

> **날짜**: 2026-04-03
> **태그**: `OAuth`, `보안`, `버그수정`, `버그바운티`
> **기여**: 임서원 (버그바운티 제보)

## 무엇을 했나요?

1. OAuth Bearer 토큰으로 기존 API(wallet, posts 등)에 접근할 수 없던 버그를 수정했습니다.
2. OAuth 스코프 기반 API 접근 제한을 적용했습니다.

## 버그바운티 기여

이 수정은 **임서원** 학생이 [4주차] OAuth 연동 버그바운티 과제에서 제보한 버그입니다.

임서원 학생은 "엄마맘" (AI 식사 관리 앱)을 개발하면서 EarnLearning OAuth를 연동하는 과정에서, `read:wallet` 스코프를 허용했음에도 지갑 API에 접근할 수 없는 문제와, `read:posts` 스코프를 허용했음에도 게시물 데이터를 받지 못하는 문제를 발견하여 상세한 재현 방법과 함께 제보해주었습니다.

이 외에도 에러 메시지 개선, API 문서 응답 예시 추가 등 유용한 개선 제안도 함께 제출했습니다.

## 왜 필요했나요?

OAuth 인증 시스템은 두 가지 토큰 방식을 사용합니다:
- **JWT 토큰**: 직접 로그인한 사용자 (이메일/비밀번호)
- **OAuth Bearer 토큰**: 외부 앱을 통해 인증한 사용자

문제는 기존 API들이 JWT 인증만 허용하고 OAuth Bearer 토큰을 인식하지 못했습니다. 또한 OAuth 스코프(권한 범위)가 정의만 되어 있고 실제로 적용되지 않아, 어떤 스코프로든 모든 API에 접근 가능한 보안 문제가 있었습니다.

## 어떻게 만들었나요?

### 1. JWT-OAuth 인증 통합

`JWTAuth` 미들웨어에 OAuth fallback을 추가했습니다:

```
요청 → Bearer 토큰 추출
         ↓
    JWT 파싱 시도 → 성공 → 기존처럼 진행
         ↓ 실패
    OAuth 토큰 검증 → 성공 → user_id, role, status, scopes 설정
         ↓ 실패
    401 Unauthorized
```

### 2. 스코프 기반 접근 제한

`RequireScope` 미들웨어를 수정하여:
- JWT 유저 → 스코프 제한 없이 통과 (기존과 동일)
- OAuth 유저 → 해당 API에 필요한 스코프가 있는지 확인

각 API에 적절한 스코프를 매핑했습니다:
- `read:wallet` → 지갑 조회, 대출 조회
- `write:wallet` → 송금, 대출 신청
- `read:posts` → 게시글/댓글 조회
- `write:posts` → 게시글/댓글 작성, 파일 업로드
- `read:company` → 회사 정보 조회
- `read:market` → 프리랜서/투자/거래소/과제 조회
- `read:notifications` → 알림 조회

## 배운 점

- **인증 방식의 공존**: 하나의 서비스에서 여러 인증 방식(JWT, OAuth)을 지원할 때, 모든 보호 라우트에서 두 방식이 모두 동작하는지 확인해야 합니다.
- **스코프는 정의만으론 부족**: 스코프를 정의(ValidScopes)하고 UI에서 보여주더라도, 실제 API 엔드포인트에 적용하지 않으면 보안적으로 의미가 없습니다.
- **버그바운티의 가치**: 개발자가 놓치기 쉬운 실제 사용 시나리오의 버그를 외부 테스터가 찾아줄 수 있습니다.
