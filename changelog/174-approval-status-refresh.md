# "승인했는데 왜 아직 대기중이래요?" — 오지 않는 이벤트를 기다린 화면

## 무엇을 했나요?

관리자가 가입을 승인해도 학생 화면에는 계속 "관리자 승인을 기다리고 있습니다"가 떠서 로그아웃 후 재로그인해야 했던 문제를 고쳤습니다. 이제 승인되면 **5초 안에 자동으로** 메인 화면으로 넘어갑니다. 승인 알림("가입 승인 완료")도 새로 생겼습니다.

## 왜 그랬나요? — 세 겹의 원인

수업 중 다수 제보로 파악한 원인은 하나가 아니라 셋이었습니다:

1. **이벤트가 발송되지 않았습니다.** 대기 화면은 `user_approved` 웹소켓 이벤트를 기다리는데, 승인 처리 코드는 상태만 바꾸고 이벤트를 어디에도 안 보냈습니다.
2. **보냈어도 못 받습니다.** 웹소켓 서버가 "승인된 유저만 연결 가능"이라 대기 중인 학생은 연결 자체가 거부됩니다.
3. **토큰에 상태가 박제되어 있습니다.** 로그인 토큰(JWT)에 `status: pending`이 새겨져 있어, DB에서 승인돼도 서버는 옛 토큰을 계속 대기자로 취급합니다. 그래서 재로그인(새 토큰 발급)이 유일한 탈출구였던 것.

## 어떻게 고쳤나요?

핵심 발견: 이미 있던 `POST /auth/refresh`가 완벽한 해답이었습니다 — 옛 토큰을 받아 **DB의 현재 상태로 새 토큰**을 발급해줍니다. 대기 화면이 5초마다(+ 탭에 돌아올 때 즉시) 이 엔드포인트를 폴링하고, 승인이 확인되면 새 토큰을 저장한 뒤 **전체 페이지 리로드**로 앱을 재시동합니다. 리로드 덕에 웹소켓도 승인된 토큰으로 재연결되어 실시간 기능까지 즉시 살아납니다.

부수로 승인 시 알림을 발송하도록 추가했습니다 (알림함 + 계정 이메일).

## 배운 점

- "이벤트 기반 UI"는 이벤트 발송·전달·수신 세 구간을 모두 검증해야 합니다. 한 구간이라도 끊기면 조용히 영원히 기다립니다. 폴링은 촌스러워 보여도 끊길 구간이 없습니다.
- 인증 토큰에 가변 상태(승인 여부)를 넣으면 그 상태가 바뀔 때의 갱신 경로를 반드시 설계해야 합니다. refresh 엔드포인트가 이미 있었는데 아무도 안 쓰고 있었습니다.
